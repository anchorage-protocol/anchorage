# PRD

> The technical north star: data model, interfaces, governance machinery, calibration, credit, adversary testbed. Where the [manifesto](./manifesto.md) explains *why* and [governance](./governance.md) sketches *who decides what*, this document specifies *how*.

This document captures design intent. **Field-level contracts** (zod schemas, MCP tool I/O shapes, lifecycle states) live in [`@anchorage/contracts`](../packages/contracts/src) and are the executable spec. The two stay in lockstep under the docs-never-drift discipline (see [CLAUDE.md](../CLAUDE.md)) — a contract change without a doc change, or vice versa, is a bug. Parameters deliberately deferred to the adversary testbed — calibration ratios, reputation decay rates, vote thresholds — remain unspecified here; they are tuned against simulation rather than guessed, with Phase 1's cube suite ([docs/phase1-results.md](./phase1-results.md)) reading the closure properties each parameter axis pins, and the testbed remaining the regression handle for any future change.

## Architecture overview

Anchorage is one canonical service layer behind two client surfaces:

```
                         ┌─────────────────────────────┐
                         │   Canonical service layer   │
  ┌──────────────────┐   │  ─ graph store              │   ┌──────────────────┐
  │  MCP server      │◀─▶│  ─ write-path tools         │◀─▶│  Web UI          │
  │  (mcp.…)         │   │  ─ verification engine      │   │  (anchorage.…)   │
  │  contributor /   │   │  ─ governance machinery     │   │  read & browse   │
  │  agent surface   │   │  ─ reputation & calibration │   │  for humans      │
  └──────────────────┘   │  ─ projection engine        │   └──────────────────┘
                         └─────────────────────────────┘
```

- **MCP server** (`mcp.anchorage.science`): primary write-path interface. Most traffic is agents acting as a human contributor's delegate (see [Identity](#identity)) — connected from any MCP-capable client and pulling assignments during idle time. Direct human contribution and the simulated populations in the testbed connect through the same surface. Tools are typed; verification is server-side.
- **Web UI** (`anchorage.science`): read-mostly human surface. Browse causes, sub-topics, graphs, frontiers, and manuscript projections. Calls the same canonical service layer.
- **Service layer**: the trust boundary. Every mutation passes through verification, governance gates, and reputation updates here. Clients are untrusted regardless of identity.

This is a deliberate architectural commitment, not an implementation note. Federation between Anchorage instances later is MCP-to-MCP. The testbed connects via the same MCP interface real clients use — *no stub APIs*. The architectural commitment is what the testbed depends on; the broader claim that simulated populations cover the real contributor distribution is qualified in the [manifesto](./manifesto.md#testability-is-the-secret-weapon) and the [testbed coverage section](#what-the-testbed-does-not-cover).

## Data model

### Multi-scale structure

Three layers, each with the shape suited to its job:

- **Cause** — the umbrella unit of belonging ("colon cancer"). Causes are created by maintainers; they are not user-creatable in v0.
- **Sub-topic** — scope envelope within a cause ("ctDNA-MRD in stage II resected CRC"). Sub-topics are first-class objects: they have IDs, descriptions, scope-envelope queries (e.g. PubMed search definition), creation status (proposed / active / archived), and curator approval state. In v0 sub-topic creation requires curator approval; later phases admit auto-discovery from graph state.
- **Claim graph** — the per-cause structure where work happens, partitioned across sub-topics by node *home* and connected across them by *scope memberships*.

A node has **exactly one home sub-topic** and **zero or more scope memberships** in other sub-topics within the same cause. The two are different jobs:

- The **home** sub-topic owns the node for review purposes: it determines which reviewer pool evaluates the proposal and where the contributor's per-sub-topic reputation accrues.
- A **scope membership** is a (node, sub-topic) edge stating "this node is in scope for this sub-topic." Memberships are themselves proposable, reviewable claims — small, atomic, and decomposable. A landmark trial excerpt or a definition node ("MSI-high CRC") can legitimately be in scope for several sub-topics simultaneously without being duplicated, re-fetched, or forked across supersedes chains.

This single structure replaces the older "exactly one sub-topic, with cross-links between" rule. Cross-links as a separate edge type are gone; their use case is subsumed by memberships, with stricter semantics (memberships are reviewable assertions about scope; they do not propagate `derives` lineage across sub-topics by themselves).

### Nodes

Every node has:

- `id` — opaque identifier
- `home_sub_topic_id` — the sub-topic that owns the node for review and reputation purposes
- `scope_memberships` — set of additional sub-topic IDs (within the same cause) this node is in scope for; each membership is an independently reviewable assertion
- `kind` — one of:
  - `anchor` — external source (paper, dataset, definition). Has `external_ref` (PMID, DOI, URL) that must resolve.
  - `excerpt` — tight claim tied to a specific anchor parent. Has a `quoted_span` field; the verification engine matches it against the resolved source. Excerpts cannot exist without a verified span. The `content` is the atomic claim; the `quoted_span` is the verbatim slice that anchors verification. They are not required to be identical — `content` may paraphrase to atomize a claim the span supports — but `content` must be an assertion the span supports under a charitable reading. The verification engine confirms the span resolves; the reviewer confirms `content` follows from it. Span verification is necessary but not sufficient: cherry-picking a true span out of context (negation-stripping, hedge-stripping) is a known attack and is part of the reviewer's responsibility to catch.
  - `synthesis` — explicit inferential step derived from multiple parents. The agent or contributor sets `kind=synthesis` when the content is not a straight excerpt.
  - `open_question` — scoped uncertainty with edges to what it depends on. Surfaces as a frontier item.
- `content` — the claim text (single atomic claim per node)
- `external_ref` — for anchors only, structured pointer (PMID/DOI/URL). Excerpts do not carry an `external_ref`; their grounding flows through the `derives` edge to the parent anchor, where verification rests. Duplicating the field on excerpts would create two places for it to drift.
- `content_hash` — for anchors only, hash of the fetched source content; set by the server post-fetch and used for re-verification (see [Verification engine](#verification-engine)).
- `last_verified_at` — for anchors only, wall-clock timestamp of the most recent successful verification of `content_hash` against the live source. Set to the node's `created_at` on initial verify (acceptance time), bumped by the re-verification scheduler on every fetch whose hash still matches, and frozen at the moment-last-known-good when the scheduler flips the anchor to `unresolvable` on drift. The scheduler picks the oldest `last_verified_at` first; see [Verification engine](#verification-engine).
- `quoted_span` — for excerpts, the verifiable span text + offset within the source
- `status` — `staged` (under review), `active` (merged), `superseded` (replaced), `rejected` (review failed), `unresolvable` (anchors only — re-verification has failed and the node surfaces as a frontier item; see [Verification engine](#verification-engine))
- `created_by`, `created_at`, `updated_at`

The **active node rule**: a node is *inactive* if it is the `from` end of a `supersedes` edge.

### Edges

- `derives` — parent (support) → child (derived claim). Direction matches storage. Lineage walks backward along `derives` until it hits anchors. `derives` parents must be `active` at the moment of acceptance: a child cannot be merged with a `staged` or `rejected` parent. A `derives` edge is valid when the parent and child share at least one sub-topic — either both home there, or one home and the other a scope-member, or both scope-members. Lineage chains can therefore cross sub-topic boundaries through *shared scope memberships* without being separate edges.
- `supersedes` — old → replacement. Marks the old node inactive. The `to` end (the replacement) must be `active` at the time the supersedes is proposed. Supersedes cycles (A → B → C → A) are forbidden by the verification engine.

Other edge types are rejected. The minimal vocabulary is load-bearing: more edge types create more places for governance disputes without proportionally more expressive power. Note that *scope membership* is a property of the node, not a separate edge type — a node carries its set of sub-topic memberships directly.

### Scope membership

A scope membership says: "node X is in scope for sub-topic S." It is proposable (`propose_membership`), reviewable, and revocable through the same governance machinery as any other claim — because it *is* a claim. Memberships are evaluated by reviewers from the *target* sub-topic S (the one the node is being claimed to be in scope for), since they are the ones calibrated on that sub-topic's scope.

Memberships are how cross-sub-topic concerns compose without duplication, without forking supersedes chains, and without smuggling lineage. A definition node ("MSI-high CRC") homed in a screening sub-topic can be a member of Lynch-surveillance, ctDNA-MRD, and immunotherapy-eligibility sub-topics — one node, one verification, one supersedes chain, four sub-topics that all see and review it.

**Credit accrues to the proposer at the home sub-topic only**; downstream memberships do not multiply credit. Contributors who propose memberships (claiming a node is in scope for a sub-topic they work in) accrue a separate, smaller credit on those memberships.

### Change of home

Sometimes a node is initially homed in the wrong sub-topic. `propose_change_of_home` moves the home sub-topic to a different one within the same cause, subject to curator approval. Other memberships are unaffected; the one exception is that if the new home was previously a scope membership, it is removed from the membership list — the home is implicitly in scope, so leaving it in `scope_memberships` would be a redundant duplicate. This is a real but rare operation; most apparent "wrong sub-topic" cases turn out to be membership-needed cases instead.

### Manuscript projection

A *projection* is a derived view of a sub-topic's graph plus editorial choices (section order, narrative voice, scope of inclusion). Projections are not a separate truth ledger — they are a function of (graph state, projection config). The graph is canonical; projections come and go.

The *projection config* is itself a governance artifact, not a private editorial document. Changes to a projection config — what's in scope, section ordering, which nodes are emphasized — affect which nodes are load-bearing for argument structure and therefore which contributors get credit. Projection configs are version-controlled in the graph; changes to them route through the standard governance-change CI process; authorship disputes resolve as governance changes to the projection config rather than as private negotiations.

The v0 projection is *config-implicit*: there is no separate `ProjectionConfig` graph record yet (the versioned-config-as-governance-artifact path described above is the target shape; the persisted record + governance-CI plumbing land in a later slice). The v0 implicit default config commits to a fixed shape, surfaced through the `manuscript://{sub-topic-id}` MCP resource ([Phase 2 slice 6a](../ROADMAP.md#phase-2--single-cause-public-instance-opened-2026-05-14)):

- **Scope of inclusion**: every `active` node whose `home_sub_topic_id` equals the sub-topic OR whose `scope_memberships` includes it — the same node set `subgraph://{sub-topic-id}` returns. The convex-hull substrate gets projected; staged/rejected work stays in the proposal queue.
- **Section order**: a fixed sequence — `sources` (anchors), `quotations` (excerpts), `synthesis` (synthesis nodes), `open_questions`. Order within each section is deterministic (created_at then id, except syntheses which sort by induced-subgraph derives degree descending and then created_at — the load-bearing signal made visible in the outline).
- **Narrative voice**: none. v0 surfaces each included node's own `content` verbatim. Narrative composition is a Phase 3 surface; v0 is a faithful projection of the graph, not a rewrite of it.

The wire shape (`Manuscript`, `ManuscriptSection`, `ManuscriptCitation`) lives in [`@anchorage/contracts/resources`](../packages/contracts/src/resources.ts); the handler is `server.resources.getManuscript`; the MCP-side registration in `packages/server/src/mcp.ts` mirrors the other five resources. The `ResourceName` enum and the wrapper-exhaustive test catch drift between the registered resources and the contracts-side surface (same posture as the slice-5a/5c additions).

A separate **admin surface** — not exposed as MCP tools — covers the *bootstrap* operations a fresh deployment needs before any contributor signs in: minting an identity (Phase 1: by curator action; Phase 2+: behind whatever identity-cost mechanism is set, see [Identity](#identity)), binding agent credentials to it, creating a cause, and seeding sub-topics. These stay out of the MCP surface because they bootstrap the very authentication chain MCP relies on (no curator-role identity exists yet to authorize a wire-level call to "mint the first curator"); the operator path is the `anchorage-admin` CLI against an on-disk store. The same operations are how the testbed sets up scenarios — sim and prod use the same admin surface, by the same indistinguishability rule that governs the tool surface.

The curator-only operations that act on an already-running graph (accept / reject / defer staged proposals, revoke a misbehaving identity, archive abandoned proposals, read the identity-clustering projection) *are* exposed as MCP tools — see the [Curator-only tools](#curator-only-tools) subsection below. Putting them on the wire is what lets a seated curator direct their agent (Claude Desktop, Cursor, custom client) to operate the moderation queue from the same MCP surface a contributor uses, rather than maintaining a parallel admin client; the privileged-tool concern raised by the bootstrap operations does not apply here, because the curator role is gated at the wire by the MCP layer (role-filtered tool discovery + a wrapper-level `permission_denied` refusal on every call) and the testbed's synthetic contributor populations never see the curator-tool block on their `tools/list`.

### Write-path tools

The default contribution path is *assignment-driven*: the system draws assignments from the frontier across all sub-topics in the cause (gap-closing tasks: orphan anchors needing excerpts, syntheses needing parents, contested claims needing review) with no capacity declaration and no sub-topic selection by the contributor (sub-topic-level choice would reopen the laundering vector by letting contributors cherry-pick easy sub-topics), and reputation accrues on assigned work. A contributor-initiated path exists but with weaker rep weighting (see [Reputation](#reputation)).

**Assignment**

There is no capacity declaration. A contributor holds **one assignment slot per `(identity, cause)`** — a single FIFO slot. You cannot pull a second assignment until the one you hold resolves. This is the whole availability model: the system learns you are available exactly when you ask for work, and learns you are done when your slot resolves. Starvation of the frontier is therefore not a bounded property the system tunes against — it is a non-property: no contributor, buggy or adversarial, can lock more than one frontier slot at a time, so "stop the client from over-pulling" needs no rate cap, no declared kinds, and no trust in client behavior.

**No expertise routing.** Assignment selection does *not* consider contributor expertise, and there is no mechanism by which a contributor declares or the system infers a specialty to route on. Every assignment is a corpus task — resolve a source, match a span, synthesize a stated parent set, vote on a staged proposal — that any contributor can pull. This is a premise the rest of the architecture already assumed: the agent-as-delegate model means an arbitrary contributor's agent donates idle time to a cause, an agent carries no credentialed expertise to gate on, and the entire defense against bad work is calibration, redundant review, and verifiable anchors applied *after* the work lands — never a pre-filter on who produced it. Per-`(cause, sub-topic)` reputation remains a *descriptive* trust-and-quality record of where a contributor's work has survived; it is never a router.

- **`request_assignment`** `{ cause_id, kind? }` → `{ status: 'assigned', assignment_id, task } | { status: 'idle', cause_id, reason, sub_topics, guidance }`
  - Pull a task from the frontier. Refused if the caller already holds an unresolved slot for the cause. The system selects across all sub-topics in the cause on frontier priority alone (gap urgency, sub-topic activity); `kind`, when supplied, is a strict filter on the work kind, not a soft preference and not an expertise signal. The task is concrete: a specific node-shape to propose, or a specific proposal to review, in a specific sub-topic. Pulling is the only verb — typically driven by a delegated agent during idle time — and there is no push. **The returned slot is held the moment it is returned (`accepted`) — there is no accept step.** Single-slot has no decision at offer time: with no `decline_assignment` there is nothing to accept *into*, so an `accept_assignment` call would be pure ceremony. The two-phase `offered → accepted` handshake collapses; the only states are *held* and *resolved*.
  - **Frontier exhaustion is an honest result, not an error.** When the graph-derivable frontier has nothing eligible for this caller (review work for others' staged proposals filtered by conflict-of-interest / already-voted / cross-stratum gates, gap-closing tasks over accepted nodes, etc.), the call returns `status: 'idle'` carrying the cause's active sub-topics (each with its `scope_query`) and a `guidance` string pointing at the spontaneous-proposal path. The cause is still open: the `propose_*` tools remain callable off-slot (this caller holds no slot, having just been refused one, so the "contributor-initiated work is off-slot" guard does not fire), and that path produces the inventory the *next* contributor's `request_assignment` will draw from. This closes the cold-start UX gap a small-population live deployment surfaced — "no scheduled work" was previously a `not_found` error a freshly-connected agent read as "stop," but the design intent was always "switch modes." The `guidance` is **not fixed**: when the caller has already reviewed still-staged proposals in this cause, the idle state is the queue *waiting on other contributors' independent votes*, not a caller-side dead end (the live four-contributor cold-start trap: every staged item sat one independent vote short, and a friend's agent that correctly reviewed all of them saw zero acceptances with no explanation). In that case the prose says so and dampens the propose-more reflex, since fresh proposals also need independent reviewers before they converge — so piling them on does not clear what is already pending. The branch is **qualitative only**: `reason` and the payload *shape* are identical either way, and no vote counts or convergence-proximity are exposed — that would hand a coalition the timing to land a closing vote — while the caller already knows which proposals it voted on, so the branch leaks nothing new. The reputation-gate refusals (`assignment_min_recent` / `assignment_min_demonstrated`) deliberately stay `not_found` errors: their remedy is bootstrapping via contributor-initiated voting, not spontaneous proposing, so an idle response with the same guidance would mislead.
  - There is no `decline_assignment` and no decline state. Every reason a contributor might have declined is handled without one: *not wanting this work* is impossible to express selectively, because the single slot forbids shopping — you do not progress until this slot resolves, so there is nothing to decline *into*; *lacking expertise* is not a category, since no work is expertise-gated; *the task cannot be honestly completed* (a source that will not resolve, a parent that went `unresolvable`, a proposal already curator-resolved) is not a refusal but **the assignment's correct result** — see precondition-lapse below.
  - **Contributor-initiated work is off-slot and refused while a slot is held.** The contributor-initiated path — a `propose_*` or `cast_review_vote` call with no `assignment_id` — is not a parallel work channel. While the caller holds an unresolved slot in that cause it is refused with `invalid_state` pointing at the held slot (echo its `assignment_id` to fulfill it, or let it resolve first). This is what makes "there is nothing to shop into" an *enforced* property rather than an aspiration: without it a held-slot contributor could drive unbounded contributor-initiated proposals as a side channel that defeats the one-slot guarantee, and a reviewer that omitted `assignment_id` on an assigned review would silently strand its sole slot with no recovery — review tasks neither lapse on target resolution nor take TTL-shadow, so the strand was a dead-end (caught by the model-backed testbed before it shipped). The guard is cause-scoped: single-slot is per `(identity, cause)`, so a slot held in another cause does not block contributor-initiated work here. The genuinely contributor-initiated path (caller holds no slot in the cause — e.g. a no-reputation contributor bootstrapping eligibility) is unaffected.
  - **Precondition-lapse resolves the slot, not the contributor.** An assignment whose precondition no longer holds auto-resolves with no contributor action and no penalty: the slot closes and frees, and where the lapse is itself a finding — an anchor whose `external_ref` will not resolve — the resolution is recorded as a *fulfillment* carrying that honest negative result, feeding the same `unresolvable` machinery a re-verification flip would (lineage continues via `propose_supersedes` from a fresh `external_ref`). "Unresolvable-as-fulfillment" is the anchor instance of this general rule; the rule covers every kind (an excerpt under a now-`unresolvable` anchor, a synthesis over a superseded parent, any task whose sub-topic or cause closed). Credit accrues to whatever fulfillment materialized the slot; a contributor whose slot lapsed through no fault takes no reputation hit and no credit — symmetric with not having delivered it.
  - **TTL is shadow-reassignment, never a contributor-steered exit.** A slot idle past its TTL is not expired out from under its holder; instead the same task is *additionally* offered to a backup contributor. The slot resolves on the first accepted fulfillment by *anyone* — original holder or shadow — and the holder is released the moment their slot resolves regardless of who delivered it; a later duplicate fulfillment of an already-resolved slot is dropped and credited nothing. If both holder and shadow stall, TTL fires again onto a third. Liveness is thus guaranteed without ever kicking anyone, and — because the single-slot rule means letting TTL lapse gains the holder nothing (their one slot stays occupied either way) — TTL creates no selection channel and is not a covert decline. A permanently-dead client simply parks its own slot harmlessly; no curator-tunable expiry sweep is needed for liveness (any hard reaping of long-dead slots is a state-growth concern, not an incentive or liveness one).

**Proposals.** Each `propose_*` tool below takes an optional `assignment_id`. When present, the call **fulfills** the named accepted propose-kind assignment: the assignment's task kind must equal the proposal kind and the task's pinned context (sub-topic, parent anchor, parent set) must match what the proposal claims, the assignment transitions to `submitted` with this proposal as `fulfilled_by`, and the work accrues full assigned-work reputation. When absent, the proposal is **contributor-initiated** and weighted lower for reputation (see [Reputation](#reputation)) — and, under single-slot, refused outright while the caller holds an unresolved slot in that cause (see [Assignment](#assignment), "Contributor-initiated work is off-slot"). This mirrors `cast_review_vote`'s optional `assignment_id` for review work — there is one tool per node kind whether or not an assignment is being fulfilled, rather than a separate assignment-only entry point. (`propose_change_of_home` and `propose_sub_topic` are curator-gated and not assignment-driven, so they carry no `assignment_id`.)

- **`propose_anchor`** `{ cause_id, home_sub_topic_id, memberships?, content, external_ref, assignment_id? }` → `{ proposal_id }`
  - Creates a staged anchor node. Server fetches `external_ref`, confirms resolution, and rejects on failure. `memberships` is an optional list of additional sub-topic IDs in the same cause; each becomes a separately-reviewable membership claim.
- **`propose_excerpt`** `{ cause_id, home_sub_topic_id, memberships?, parent_anchor_id, content, quoted_span, assignment_id? }` → `{ proposal_id }`
  - Creates a staged excerpt node. Server matches `quoted_span` against the resolved source. Mismatch → rejection. No exceptions.
- **`propose_synthesis`** `{ cause_id, home_sub_topic_id, memberships?, parent_ids, content, kind, assignment_id? }` → `{ proposal_id }`
  - Creates a staged synthesis or open_question node with `derives` edges from each parent. Atomic — either all edges create or none do. `kind` is `synthesis` or `open_question`.
- **`propose_supersedes`** `{ from_node_id, to_node_id, rationale, assignment_id? }` → `{ proposal_id }`
  - Stages a supersedes edge with the reasoning attached.
- **`propose_membership`** `{ node_id, sub_topic_id, assignment_id? }` → `{ proposal_id }`
  - Stages a scope-membership claim that `node_id` is in scope for `sub_topic_id`. Reviewed by the *target* sub-topic's reviewer pool.
- **`propose_change_of_home`** `{ node_id, new_home_sub_topic_id, rationale }` → `{ proposal_id }`
  - Curator-approved.
- **`cast_review_vote`** `{ proposal_id, decision, rationale, assignment_id? }` → `{ vote_id }`
  - `decision` is `accept`, `reject`, or `revise`. `rationale` is required and may itself be promoted to a graph node (typically `open_question`) by curators; promoted rationale-nodes pass standard review. When `assignment_id` is present, the vote fulfills a review assignment and accrues full assigned-review reputation; without it, the review is contributor-initiated and weighted lower on the same terms as contributor-initiated proposals — and likewise refused while the caller holds an unresolved slot in the proposal's cause (see [Assignment](#assignment), "Contributor-initiated work is off-slot"); a reviewer fulfilling an assigned review **must** echo its `assignment_id`. This tool is therefore the fulfillment path for review-kind assignments as well as the contributor-initiated review entry point.
- **`propose_sub_topic`** `{ cause_id, name, description, scope_query }` → `{ proposal_id }`
  - Subject to curator approval in v0.

### Read-path tools and resources

Read-path is largely MCP *resources* (passive), with a few active tools for queries:

- **Resource: `cause://...`** — list of active causes with their active sub-topics; the passive twin of the `query_causes` tool (identical `CauseDirectory` shape).
- **Resource: `sub-topic://{id}`** — sub-topic metadata, status, scope query, recent activity.
- **Resource: `node://{id}`** — node + immediate neighbors.
- **Resource: `subgraph://{sub-topic-id}`** — full or filtered subgraph in a structured form.
- **Resource: `contributor://{id}`** — public contributor profile + per-(cause, sub-topic) eligibility-tier projection (see [Reputation](#reputation) for the tier definition).
- **Resource: `manuscript://{sub-topic-id}`** — manuscript projection of a sub-topic: outline (sources, quotations, synthesis, open questions) + credited contributors (see [Manuscript projection](#manuscript-projection) for the v0 implicit-config shape and [Credit](#credit) for the credit function).
- **Tool: `query_causes`** `{}` → `CauseDirectory` (active causes + their active sub-topics). The agent's bootstrap entry point and the tool-surface twin of the `cause://` resource — same shape, one implementation (`server.resources.getCauseDirectory`). It exists because the *tool* surface must be self-sufficient for the first contribution: `request_assignment` requires a `cause_id`, and an MCP client is not guaranteed to surface resources to the model driving it (many expose only tools, or gate resources behind explicit user attach). Without a tool, a freshly-connected agent that just completed OAuth has no in-band path from "I want to contribute" to a `cause_id` — the worst place to lose a new contributor. Resources stay the canonical passive browsing mirror; this is the active enumeration the bootstrap needs.
- **Tool: `query_frontier`** `{ cause_id?, sub_topic_id?, frontier_kind? }` → ordered list of frontier items (work to be done).
- **Tool: `query_proposals`** `{ status?, sub_topic_id?, assigned_to_me? }` → list of proposals matching filter.
- **Tool: `fetch_calibration_batch`** `{ sub_topic_id }` → calibration-material projection: accepted-from-validated-history proposals routed to the sub-topic, in [`ReviewBatchItem`](../packages/contracts/src/tools.ts) shape (`proposal_id` + `payload`, status/proposer/timestamps stripped). The reviewer-facing calibration path runs through `request_assignment` (see [Calibration batches](#calibration-batches)) — a calibration item is delivered as a review-kind assignment indistinguishable from real review work, scored on `cast_review_vote` against ground truth. `fetch_calibration_batch` is the diagnostic projection of the same pool: the testbed inspects it; production reviewers do not call it directly.
- **Tool: `query_reputation`** `{ cause_id }` → `{ entries: [{ sub_topic_id, demonstrated, recent }] }`. The caller's *own* per-(cause, sub-topic) reputation, both components decayed forward to the server's current time on read. PRD §Reputation commits "Eligibility tiers public; numeric reputation private" — the contributor sees their own raw numbers (otherwise they cannot reason about where they sit relative to tier gates and which assignment-gate thresholds will fire against them), but other contributors see only tiers via a separate read-path surface that will land when tiers are defined. Returning the decayed live values rather than the at-last-bump snapshots is what makes the recent-activity gate's quiet-window arithmetic visible to the contributor; without it the patient-adversary signature (drained `recent`) would be observable to the server but invisible to the caller.

Tool surface is intentionally small. Each tool has tight typing, server-side validation, and clear failure modes.

*Implementation status:* the five query tools above are wired in v0 (ROADMAP §Status: all 14 contributor-facing tools from the PRD, including the read-path projections `query_causes`, `query_frontier`, `query_proposals`, `fetch_calibration_batch`, `query_reputation`). `query_causes` shares its single implementation with the `cause://` resource (`server.resources.getCauseDirectory`) — passive resource and active tool cannot drift because there is one method behind both. The first four resources (`cause://`, `sub-topic://{id}`, `node://{id}`, `subgraph://{sub-topic-id}`) are wired in slice 5a of the [Phase 2 plan](../ROADMAP.md#phase-2--single-cause-public-instance-opened-2026-05-14); the fifth (`contributor://{id}`) is wired in slice 5c alongside the public eligibility-tier projection; the sixth (`manuscript://{sub-topic-id}`) is wired in slice 6a alongside the v0 implicit-default projection config and the credit function — `server.resources.{getCauseDirectory, getSubTopicDetail, getNodeNeighborhood, getSubgraph, getContributorProfile, getManuscript}` in `packages/server/src/server.ts` produce the structured payloads, and `packages/server/src/mcp.ts` registers them through the SDK's `registerResource` surface with `application/json` content blocks. The wire shapes live in [`@anchorage/contracts/resources`](../packages/contracts/src/resources.ts): `CauseDirectory` is the home-page payload (active causes with their active sub-topics, stable-ordered); `SubTopicDetail` carries metadata + a `SubTopicActivityCounters` projection (active-node count, staged-proposal count, frontier-item count) derived from current state; `NodeNeighborhood` returns the node + its immediate active edges + hydrated neighbors; `Subgraph` returns the active node set (home OR scope-member) with edges restricted to that node set; `ContributorProfile` returns `PublicContributor` (display fields only — no IdP subject, attestation level, role, or provider) plus a `PublicReputation.entries` array of tier-labeled per-(cause, sub-topic) records (no raw numeric reputation, see §Reputation for the tier mapping); `Manuscript` returns the sub-topic + cause + a fixed-order `ManuscriptSection` array (sources, quotations, synthesis, open_questions) + a `CreditAttribution` list ordered by descending units (each entry carrying the contributor's id, public display fields, units, and proposed/reviewed-node breakdown). Caller gating matches the read-path tools — the token must resolve through the Authenticator, but resource reads do not consume the per-identity rate-limit budget. `ResourceName` is the contracts-side enum the harness can introspect against; the in-test exhaustiveness assertions (contracts-side enum vs. wrapper-side registration) catch drift the same way the `ToolName` test does.

#### Agent bootstrap

The first-use path after a client is added and OAuth completes is: **`query_causes` → pick a `cause_id` → `request_assignment` for it → fulfill the offered task with the matching `propose_*` tool or `cast_review_vote`. When `request_assignment` returns `status: 'idle'` rather than an assigned slot, the cause is still open — switch to spontaneous proposing, lowest-friction first: connect accepted nodes via `propose_synthesis` or supersede stale anchors via `propose_supersedes` (the agent already saw the accepted graph through review work and can browse subgraph:// for more), and only then `propose_anchor` for in-scope literature not yet anchored (each returned `sub_topic.scope_query` is the guide).** This is the load-bearing UX seam — the moment a freshly-authenticated contributor either starts or bounces — so it is closed two ways, both on the tool surface (no dependency on a client surfacing resources):

- **`query_causes`** is the in-band entry point (see the tool bullet above for why a tool, not only the `cause://` resource).
- The MCP server sets an initialize-time **`instructions`** string (the `ServerOptions.instructions` the SDK returns in the `initialize` result; clients inject it into the model's context before any tool call) carrying the sequence above in one short paragraph, including the idle-switch-modes branch that closes the cold-start trap a small-population live deployment surfaced. The per-step detail lives in the tool descriptions; the instructions string is the orientation that gets an agent to the *first* call without trial-and-error. `request_assignment` refuses an unknown `cause_id` with `not_found` and a message naming `query_causes`, so a wrong guess self-corrects.

`instructions` is a public MCP surface and is set in `buildMcpServer` (`packages/server/src/mcp.ts`); it is identical for every caller (no per-role or per-identity variation — the curator-tool *discovery* filter is the only role-conditioned part of the wire, and it does not touch `instructions`), preserving sim≡prod by construction.

#### Anonymous-browse surface

`anchorage.science` (the web read-UI introduced in [Phase 2 slice 5b](../ROADMAP.md#phase-2--single-cause-public-instance-opened-2026-05-14)) renders the read-path resources above as HTML for anonymous browsers. The anonymous-browse posture is a single operator-minted *service caller* — an `anchorage-admin mint-reader`-issued contributor-role harness identity that the web tier holds for the lifetime of the deployment. Anonymous web users get HTML pages from the web service; they never present a bearer themselves. The earlier sketch of "one anonymous identity minted per browse session" is not what slice 5b implements: per-session minting would multiply identity count for no gain (resource reads do not consume per-identity rate-limit budget, see the gating note above) and would press needlessly against the IdP's binding-cost gate. Auth-required contributor views (a later slice) still mint per-human identities through the OAuth flow; this section commits the anonymous shape only.

The web tier is co-located with the MCP server (`run-prod.ts` mounts both surfaces on the same `node:http` listener), so the reader is an in-process method-dispatch wrapper around `server.resources.*` and `server.tools.queryFrontier`. The composition is opt-in: omit `ANCHORAGE_WEB_READER_IDENTITY` and the deployment stays MCP-only, with `/` and `/sub-topic/*` returning the typed-JSON 404 from the MCP HTTP transport rather than HTML. A future federated-peer deployment that wanted the web tier on a separate process would swap `InProcessReader` (in `@anchorage/server`) for an MCP-over-HTTP-backed implementation; the `AnchorageReader` interface in `@anchorage/web` is the seam.

### Curator-only tools

A third block of MCP tools sits behind the `'curator'` role gate. The seated-curator path (the human-or-humans whose `Identity.role` was set to `'curator'` by the operator-only bootstrap, see [Identity](#identity) and the admin-surface paragraph above) drives the [contribution-flow Resolve step](#the-contribution-flow), the [reviewer-assignment step-4 escalation](#reviewer-assignment), and the [divergence-closure archival](#reviewer-assignment) sweep that reclaims wedged proposal state. (Wedged *assignment* state no longer needs a curator sweep: a stale assignment slot is reclaimed automatically by TTL shadow-reassignment, see [§Assignment](#assignment).) All of it runs on the wire through the same MCP surface a contributor uses.

- **`curator_accept_proposal`** `{ proposal_id }` → `{ node_id?, sub_topic_id? }` — accept a staged proposal; materializes whichever underlying node / edge / sub-topic the proposal kind dictates. Returns whichever id the proposal materialized (a node_id for graph-creating kinds, a sub_topic_id for sub_topic, neither for in-place mutations like membership / supersedes / change_of_home).
- **`curator_reject_proposal`** `{ proposal_id }` → `{ ok }` — close a staged proposal as rejected without materializing it. Rep-neutral curator override (a curator tiebreak is not the peer agreement the reputation ledger credits).
- **`curator_defer_sub_topic`** `{ proposal_id }` → `{ sub_topic_id }` — defer a staged `sub_topic` proposal: materialize as a `proposed` (not `active`) `SubTopic`. The defer-as-proposed path PRD §Sub-topic creation commits.
- **`curator_revoke_identity`** `{ identity_id }` → `{ identity_id, status, changed }` — flip an identity to `revoked`. Idempotent: re-revoking returns `changed: false` rather than refusing. Same revocation-cascade semantics as the `anchorage-admin revoke-identity` CLI (the CLI now delegates to this surface so the on-the-wire and offline paths cannot drift).
- **`curator_archive_stale_proposals`** `{ window_seconds, cause_id? }` → `{ proposal_ids }` — divergence-closure sweep. Transitions staged proposals whose most recent vote is older than `window_seconds` to `unresolved-archived`. The trigger cadence is operationally private; testbed-tunable.
- **`curator_identity_clusters`** `{ window_seconds?, min_signal? }` → `{ pairs: [{ identity_a, identity_b, cross_cause_count, shared_proposal_count }] }` — cross-cause identity-clustering projection over shared-proposal vote co-occurrence. The fourth of the four sybil-resistance layers per [Identity](#identity); the curator decides what counts as a coalition vs. coincidence.
- **`curator_reverify_anchors`** `{ batch_size, max_age_ms, cause_id? }` → `{ checked, unchanged, unresolvable, anchors: [{ anchor_id, outcome }] }` — batch re-verification of `active` anchors. Re-fetches each anchor's `external_ref` through the configured `Verifier`, oldest-`last_verified_at` first, and either bumps `last_verified_at` (hash match — "still known good") or flips the anchor to `unresolvable` (hash mismatch, or the verifier itself refused — retraction, host gone, network failure all collapse here; the verifier seam does not distinguish them). The production runtime ticks this on a scheduler; operators can also drive it on-demand. `unresolvable` is terminal for the anchor; the lineage continues via `propose_supersedes` from a contributor proposing a fresh `external_ref`.

**Authorization shape.** Two layers — discovery filter and wire-level gate — both running off the same `Identity.role` field:

- *Discovery filter* (`buildMcpServer` in `packages/server/src/mcp.ts`): the `curator_*` block is only registered on the MCP session when the resolved caller's role is `'curator'`. A contributor's `tools/list` response omits the block entirely, matching what they could call. This is what keeps the testbed's contributor-archetype request bodies byte-identical to pre-slice-7a (the LLM-backed archetypes' Anthropic-API request shape is a function of `mcpClient.listTools()`; hiding curator tools from non-curator sessions means pre-existing cassettes replay unchanged).
- *Wire-level gate* (`wrapCurator`, same file): every `curator_*` tool call re-resolves the caller from the store on invocation and asserts `identity.role === 'curator'`. A demotion that lands mid-connection (rare in v0 — role is admin-only via the `anchorage-admin` CLI — but in principle possible) refuses the next call with the typed code `permission_denied`. The discovery filter is a hint; this is the authoritative gate.

`permission_denied` is a typed `ServerErrorCode` (in [`@anchorage/contracts`](../packages/contracts/src/errors.ts)) distinct from `unauthorized` — the latter means "the token did not resolve to an active identity at all," the former means "the caller is authenticated but the role does not permit this tool." Recovery paths diverge (re-authenticate vs. obtain a role promotion from an operator), and conflating them would mislead a contributor whose signin is fine into re-authenticating. The HTTP transport maps the codes 401 vs 403.

The MCP surface is the *only* curator-action surface on the wire. The web tier (slice 5b's [Anonymous-browse surface](#anonymous-browse-surface) and slice 7b's [Curator console](#curator-console)) stays read-only; the curator clicks through the moderation queue and then directs their agent to fire `curator_*` over MCP. This matches the agent-as-delegate framing — humans direct agents that interact via MCP; the web is for reading. (`anchorage-admin revoke-identity` exists as an offline operator path too — for the case where the curator's bearer is itself compromised — but for routine moderation, MCP is the path.)

*Implementation status:* the seven `curator_*` tools above are wired across slices 7a and 7c part 1 of the [Phase 2 plan](../ROADMAP.md#phase-2--single-cause-public-instance-opened-2026-05-14); the in-process `server.curator.*` namespace was already implemented before the wire-level path landed (the testbed and the admin CLI both call it directly), so 7a is purely the wire-layer wrap plus the role-gate, and 7c part 1 adds `curator_reverify_anchors` plus the curator-side `getCuratorUnresolvableAnchors` projection. The curator-console UI that surfaces the moderation queue, the identity-cluster coordination projection, and the unresolvable-anchor list to a human reviewer is slices 7b (queue, identity clusters) and 7c part 2 (unresolvable anchors); the latter also wires the production scheduler that ticks `reverifyDueAnchors` periodically.

#### Curator console

A read-only curator-facing companion to the public web tier, mounted under `/curator/*` on the same `node:http` listener `mcp.anchorage.science` / `anchorage.science` already share. PRD §Curator-only tools commits the action surface on MCP; the console is the *browse* surface a seated curator opens to know what's pending: moderation queue (every `staged` proposal, oldest first, optional `cause_id` filter), cross-cause identity-clusters table, unresolvable-anchors list (slice 7c — anchors the re-verification scheduler flagged, most-recent-drift-first, optional `cause_id` filter). No accept / reject / revoke / re-anchor buttons on any page — actions stay on MCP, fired through the curator's agent (Claude Desktop, Cursor, custom client). The split-action posture is what keeps the web tier safe to ship without per-request authentication: the privileged operation is the agent's MCP call, not a click in the browser, so the *value* a console exfiltration could yield is the projection data itself rather than a write capability.

The console is wired by configuration through `ANCHORAGE_WEB_CURATOR_IDENTITY` (see [docs/deploy.md](../docs/deploy.md), §"Bootstrap: enabling the curator console"). Without the env, `/curator/*` 404s by route absence — a deployment that did not opt in cannot leak curator-side data by accident. With it, the in-process web handler holds a curator-role caller for the lifetime of the deployment and reads through `server.resources.{getCuratorQueue, getCuratorIdentityClusters}`, each of which `requireCurator`-gates on every call so a mid-flight revocation (or role demotion) propagates as `permission_denied` → 403 on the next request without a restart. The boot path additionally validates that the configured identity exists, is active, and holds the curator role — a contributor-role mint passed by mistake refuses at startup rather than per-page.

**Network gating is operator-side, not Anchorage-side.** The console contains operationally-private data (cross-cause identity-pair signals) and the v0 runtime exposes no login flow for the `/curator/*` routes. The operator is responsible for restricting which network origin reaches the mount-point — reverse-proxy ACL, basic auth at the edge, Cloudflare Access policy, VPN-only egress, or whatever the deployment's network primitives offer. This is a deliberate scope decision: a session/cookie auth layer adds infrastructure (session storage, login form, CSRF on a write surface that doesn't exist here) for a value proposition the agent-as-delegate model has already moved off the browser. A later phase may add browser-facing auth if the operational pattern grows past what reverse-proxy gating handles; v0 stays at this seam.

## Verification engine

The verification engine is the security boundary. Every write tool routes through it.

- **Anchor verification.** External references must resolve. PMIDs hit NCBI E-utilities; DOIs resolve via Crossref; URLs must return 200 with substantive content. Anchors are *content-addressed*: the hash of the fetched content is stored alongside the `external_ref`, and re-verification compares against the stored hash rather than only against a live fetch. URL-anchors are second-class — metadata-unstable and cloaking-prone — and may be subject to stricter regimes (or refused entirely in v0). When re-verification fails (retraction, content drift, host gone), the anchor moves to an `unresolvable` status and surfaces as a frontier item rather than silently rotting. *Implementation status:* the initial-verification path is wired as `LiveFetchVerifier` (`packages/server/src/live-fetch-verifier.ts`) — PMID via NCBI E-utilities `efetch` returning the abstract text, DOI via the Crossref `works` endpoint composing title + JATS-stripped abstract, URL refused by default in v0 (`reject_url` opt-in for curator-vetted sources). Content hashing is sha256 over the normalized source. The verifier is wired as the default `Server` verifier in the production runtime (slice 4c). The re-verification *primitive* lands in slice 7c (part 1) as `server.curator.reverifyAnchor(anchor_id)` and the batch helper `server.curator.reverifyDueAnchors({ batch_size, max_age_ms, cause_id? })`, surfaced over MCP as `curator_reverify_anchors`: each call re-fetches an `active` anchor's `external_ref`, compares the fresh content hash to the stored one, and either bumps the anchor's `last_verified_at` (hash match — keeps `active`, `updated_at` unchanged because a verification heartbeat is not a user-meaningful state change) or flips the anchor to `unresolvable` (hash mismatch, *or* the verifier itself refusing the fetch — retraction, host gone, network failure; the verifier seam does not distinguish them, see the comment in `LiveFetchVerifier.fetchPmid`). On the flip, `updated_at` moves (it is a meaningful state change the curator surface keys on) but `last_verified_at` stays put — the timestamp continues to record when the source was last known good, which is the curator's relevant question. The `getCuratorUnresolvableAnchors({ cause_id? })` projection surfaces flagged anchors most-recent-drift-first. `unresolvable` is terminal for this anchor: recovery is via `propose_supersedes` from a contributor proposing a fresh `external_ref` pointing at the same claim, not by retrying the same anchor (which would re-evaluate the same now-failing fetch and offer no recovery). Slice 7c part 2 wires the *periodic scheduler* in `run-prod.ts` (opt-in via `ANCHORAGE_REVERIFY_INTERVAL_MS` plus the `MAX_AGE_MS` / `BATCH_SIZE` companions — the three travel together; setting any without the interval refuses at boot, because there is no silent default for a cadence that drives real fetches against upstream verifiers) and the `/curator/unresolvable` web page that renders the projection for a seated curator. Per-tick errors are caught and surfaced through the structured-log sink (`anchorage.reverify.error`) so a transient upstream failure does not kill the scheduler; outcomes are logged as `anchorage.reverify.tick` with `{ checked, unchanged, unresolvable }`.
- **Span verification.** For excerpts, the `quoted_span` must be a substring of the fetched source after normalization (whitespace, quote-style, and a small set of typographic equivalences specified in the verification spec, not left to "light normalization" hand-waving). Failure rejects the proposal at write time, not at review time. Span verification confirms the quote exists; it does not confirm the proposed `content` follows from it — that is the reviewer's job. *Implementation status:* normalization is wired as `normalizeForSpanMatch` in `packages/server/src/verifier.ts`, shared between `LiveFetchVerifier` and the testbed `FakeVerifier` so the source-fixture and the production fetcher agree on what counts as a match. Three layers: smart-quote fold to ASCII straight quotes (single and double, including low-9 and prime forms); typographic fold for en-dash / em-dash / minus-sign to ASCII hyphen and horizontal-ellipsis to three dots; whitespace collapse (which also folds non-breaking spaces via the JavaScript `\s` class). The `LiveFetchVerifier` caches the fetched source in-process under the ref so that an excerpt proposed against an anchor proposed in the same session skips a second fetch.
- **Lineage validation.** A `derives` edge is valid when its parent and child share at least one sub-topic membership (home or scope). At acceptance, every `derives` parent must be `active` (not `staged`, `rejected`, or `superseded`); the `to` end of a `supersedes` must be `active` at proposal time; supersedes cycles are rejected. A manuscript projection in sub-topic S walks `derives` lineage that stays within S's home + scope-member graph.
- **Reputation gates.** Some operations require minimum reputation (per-(cause, sub-topic) or per-cause). Below the threshold, proposals land staged but are not advanced into the review queue without curator action. Specific thresholds are tuned in the testbed.
- **Rate limits and abuse signals.** Per-identity rate limits on observable write actions are spec'd in [Identity](#identity)'s per-identity rate-limit accounting bullet (the new `rate_limited` refusal mode at `propose_*`, `cast_review_vote`, and `request_assignment`); suspicious patterns (sudden burst of proposals, calibration-failure clustering, behavioral fingerprints across causes) surface to the curator via the cross-cause identity-clustering projection in the same section. Specific signals and thresholds are operationally private.

## Governance machinery

### The contribution flow

The default flow is assignment-driven. The contributor-initiated flow is a special case (step 0 is skipped; rep weighting is reduced).

0. **Assignment** — the contributor requests work (`request_assignment`); the system draws a frontier task across all sub-topics in the cause on frontier priority alone and hands back one slot per `(identity, cause)`, held the moment it is returned. There is no capacity declaration, no accept step, and no decline; the single FIFO slot is the whole availability model and sub-topic selection is the system's, not the contributor's, which is what closes the rep-laundering vector. See [§Assignment](#assignment).
1. **Submit / Propose** — work is staged through the `propose_*` tools; an assigned contributor passes the assignment's `assignment_id` so the proposal fulfills it, a contributor-initiated proposal omits it.
2. **Verify** — verification engine accepts or rejects synchronously based on grounding/lineage/rate.
3. **Stage** — accepted proposals enter the review queue. Visible and citable as proposals; not part of the canonical graph.
4. **Assign reviewers** — N reviewers are drawn from the eligible pool of the home sub-topic (or the target sub-topic, for membership proposals), with calibration items mixed in.
5. **Review** — reviewers vote with rationale via `cast_review_vote`.
6. **Resolve** — convergent vote merges; divergent vote routes to a richer review path (more reviewers, curator escalation, or carrying the divergence forward as parallel synthesis nodes / `open_question`); long-unresolved divergences archive (status `unresolved-archived`).
7. **Settle** — reputation updates for the contributor and the reviewers, weighted by outcome correctness, by claim difficulty, and by whether the work was assignment-driven or contributor-initiated.

### Reviewer assignment

Reviewers are drawn from the eligible pool — contributors with sufficient per-(cause, sub-topic) reputation — by stratified random sampling. Stratification partitions the pool into *strata* (the vote-pattern co-occurrence clusters defined below) and prefers draws across strata, so a single bias zone cannot fill a proposal's reviewer slots even when the bias-aligned reviewers individually meet eligibility. Reviewers do not pre-select which sub-topics they will be drawn for; assignment is system-driven, so the pool a proposal draws from is the system's choice, not the reviewer's.

The v0 stratum primitive is the **vote-pattern co-occurrence cluster**: two reviewers fall in the same stratum when their pairwise vote agreement on shared past proposals exceeds a threshold over a rolling window. Coalition members coordinating on bias-zone proposals concentrate into a single cluster by their own behavior; honest reviewers spread across clusters by their independence. New reviewers (no shared history yet) sit each in their own singleton cluster, contributing maximally to diversity by default. Window sizes, agreement thresholds, and cross-stratum draw weights are testbed-tuned; the *signal* — vote-history clustering as the v0 stratum — is committed.

Pairwise edges are **contention-weighted**: each shared proposal contributes weight `2 * min(accepts, rejects) / total_votes` to the agreement metric, so unanimous-easy proposals contribute 0 and split-contentious proposals contribute up to 1. The pair edges only when (a) the raw shared-proposal count meets the brand-new-reviewer floor, (b) the weighted agreement ratio meets the threshold, and (c) some non-zero contention has accumulated in the shared history. Without weighting, an honest pool that happens to share unanimous priming history collapses into a single cluster, the cross-stratum draw rule fires against its own honest members, and coalition members who avoided the priming walk through the gate the honest reviewers were locked out of — the [adversary testbed](#adversary-testbed) wires that pathology end-to-end and the contention-weighted variant closes it cleanly.

Anti-correlation is a separate cluster cue alongside agreement: a pair that systematically votes the *opposite* way on shared proposals is also displaying coordinated behavior — just inversely. Treating perfect anti-correlation (1.0 disagreement ratio) as a co-stratum edge catches the decorrelating-coalition variant cleanly, since a coalition that defeats the agreement metric by voting opposite on every shared priming proposal lights up the disagreement metric instead. The testbed scenario for the decorrelating coalition closes against this defense at threshold 1.0; the threshold is testbed-tuned, with weaker thresholds catching noisier adversaries at the cost of admitting honest reviewers who happen to systematically disagree as false positives.

A mixed-strategy coalition (a 50/50 vote pattern — two agreed, two disagreed across four shared proposals) defeats either refinement *alone*: the raw mix sits at agreement 0.5 and disagreement 0.5, below threshold 1.0 in both directions. The composition closes the gap. With contention weighting, agreement on unanimous-easy items contributes 0 weight and disagreement on contentious items contributes the full weight, so the weighted disagreement ratio collapses to 1.0 and the anti-correlation edge fires. The testbed scenarios run both halves: the mixed-strategy bypass standing open with each refinement individually, and closing under the composition (contention-weighted + anti-correlation, both at threshold 1.0). The seam the composition leaves open — a coalition that avoids co-voting on any contentious item, so the pair shares zero vote-history by construction — has no in-band cluster closure; the divergence-closure mechanism instead archives the contested targets operationally at the configured window regardless of cluster signal. Adaptations that route around the in-band closure — a coalition that avoids co-voting and coordinates off-platform so members never share any encounter the cluster signal can see — remain testbed targets if they surface; calibration-item-specific agreement (contrasted against calibration record) and payload-feature vote patterns remain on file as the candidate refinements for those.

The draw rule is **prefer not-yet-represented strata first**. Within a target reviewer count N, the assignment selector visits strata in priority order, taking at most one reviewer per stratum until N strata are filled, then loosens to per-stratum saturation if the pool is exhausted. This compounds with [pool-size scaling](#adversary-testbed): in a pool that holds K+1 honest reviewers, stratification keeps a K-coalition from monopolizing the slots; in a pool too small to furnish K+1 — the small-sub-topic regime — stratification still draws *across* whatever clusters exist, and if fewer than N strata are reachable the proposal is flagged **stratification-degraded**.

Stratification-degraded propagates into convergence: a degraded proposal requires a higher distinct-reviewer count and/or a higher weighted-vote sum to converge, making the small-sub-topic case slower to close but not capturable by the pool-fits-the-coalition dynamic. The flag is visible to the contributor, factored into convergence-threshold logic (see [Calibration batches](#calibration-batches) for the weighted-vote tally), and logged for periodic audit. Specific tightened thresholds are testbed-tuned.

Reviewer pools are evaluated in this fallback ladder:

1. **Home-sub-topic rep.** Standard path. Reviewers homed in this sub-topic.
2. **Membership-sub-topic rep.** When the home pool is exhausted, draw from contributors whose work is *scope-member* of this sub-topic — they have proven legitimate stake without being homed here.
3. **Cause-rep with stratification-degraded flag.** When 1 + 2 are exhausted, draw from cause-rep contributors and flag the proposal as stratification-degraded — same flag the cross-stratum draw raises when fewer than N strata are reachable; convergence tightens identically.
4. **Curator escalation.** When even the cause-rep pool is insufficient — including the limiting case of a divergent proposal the pool stalls on with no tiebreaker — or when stratification-degraded persists across several proposals in a sub-topic, escalate to curator review. Curator review resolves the proposal directly in either direction (`curator.acceptProposal` materializes it; `curator.rejectProposal` closes it `rejected`), or carries the divergence forward as parallel nodes / `open_question`. The accept/reject override is reputation-neutral: a curator tiebreak is not the peer agreement the convergence path's reputation deltas are calibrated against, and routing curator overrides through the rep ledger would let a curator move reputation by fiat.

At sub-topic launch, both the eligible pool *and* vote-history are thin by construction, so stratification-degraded fires immediately and calibration items are drawn from the cause's validated history rather than from the sub-topic's. The doc states this explicitly so it is not mistaken for a vulnerability when reviewers notice it.

Convergence and divergence thresholds are claim-class-aware: high-stakes claim classes (e.g., quantitative effect-size syntheses) draw larger pools and tighter convergence thresholds than low-stakes ones (e.g., terminological clarifications). Specific class definitions and threshold values are testbed-tuned; the *machinery* being class-aware is a design commitment.

Divergence has a closure mechanism. Divergent proposals are routed to richer review or carried forward as parallel synthesis nodes / `open_question`, but not indefinitely: divergences without further evidence within a tunable window are archived (status `unresolved-archived`) rather than perpetually re-routed. This prevents the queue from accumulating reviewer-noise as if it were principled disagreement.

### Calibration batches

Reviewer batches contain a mix of real proposals and calibration items. Calibration items are drawn from the graph's own validated history — proposals that survived multiple confirmations and have been stable. They are intended to be statistically indistinguishable from real frontier work in *the dimensions a reviewer can act on*: a reviewer evaluating one batch should not be able to tell which items are calibration. The harder question — whether a *patient* adversary observing many batches can build a classifier on re-use frequency, age, or other batch-level signatures — is real, and the methodology actively defends against it: calibration sampling is biased toward fresh-but-validated history, items rotate aggressively, and the sampling distribution itself is part of what the testbed evaluates as an attack surface.

Calibration items arrive on the same surface as real review work — `request_assignment` returns a review-kind task whose `proposal_id` may point at a staged proposal (real review) or at an accepted-from-history proposal (calibration). The reviewer cannot distinguish the two from the task they receive. When the reviewer votes via `cast_review_vote`, the server scores the vote against ground truth (calibration items have a known outcome — `accept` is correct, `reject` is incorrect) and adjusts the reviewer's reputation accordingly. Calibration votes do not re-resolve the underlying proposal; they only meter the reviewer. This makes calibration "just another assigned task" from the reviewer's perspective and a server-side scoring channel from the system's, which is what closes the [strategic-coalition vector](#adversary-testbed) — a coalition that votes biased on real proposals also votes biased on calibration items they cannot identify, paying for the bias even when they win the convergence battle.

Reviewers who fail calibration lose reputation; the calibration corpus grows as the graph grows.

The reviewer's calibration record is also available to the convergence layer as a vote-weighting signal: a per-(cause, sub-topic) net-passes count that scales the weight a reviewer's vote carries when the server tallies convergence. Convergence requires both a minimum count of distinct reviewers (the redundant-peer-review invariant) and a minimum *weighted* sum (the bias-resistance signal); a reviewer whose calibration record went sour can hold a convergence open but cannot drive it. The [stratification-degraded](#reviewer-assignment) flag tightens both — distinct-reviewer count and weighted sum — when the assignment selector could not draw a diverse enough pool, so a proposal that lost its diversity defense recovers it through a higher convergence bar. This compounds with the rep-ledger penalty rather than replacing it: a coalition that misfires on calibration items pays once on the rep ledger and once at convergence. Calibration-aware adversaries — bias on real proposals, ground-truth on calibration items (a known PRD §Adversary-testbed target) — bypass this layer; they remain testbed work that builds on the seam, not regressions of it. A second bypass mode surfaces *intra-run*: the calibration corpus is the graph's accepted history, so any bias-aligned excerpt that converges accepted during a review session joins the corpus as a fresh-but-validated item, and recency-bias in the calibration draw then hands later coalition members items that match their bias and pass calibration — lifting their convergence weight back above the aware threshold for whatever targets they vote on after the contamination lands. The mitigation is a corpus-composition rule at the draw layer: `corpus_confirmation_depth_floor` is the minimum distinct-reviewer count an accepted-from-history item must accumulate over the vote store before it is eligible as a calibration draw. Anchors are exempt by kind — they are the ground-truth substrate, not synthesis-from-review, and counting their accumulated calibration-vote reviewers against the floor would mis-trip the gate against the substrate itself. At v0 votes_to_accept=2, floor=3 requires at least one independent confirmation past the original convergence pair before the item enters the corpus, closing the within-first-review-session bypass. The filter is enforced at both the public read-path (`fetchCalibrationBatch`) and the assignment-injection seam (`drawCalibrationTask`), so the same eligibility rule applies regardless of whether calibration arrives via the read tool or via `request_assignment`'s injection cadence. Default 0 leaves the filter inert; production thresholds are testbed-tuned. Specific weight functions and minimum-count thresholds are testbed-tuned.

**Specific calibration items in active rotation are operationally private.** Published items are burned. Methodology — including the rotation regime and the sampling-distribution defenses against batch-level correlation attacks — is fully public; specific tuning is not.

### Reviewer-as-staking

Reviewers gain reputation when they accept proposals that survive and reject proposals that get rejected by other reviewers. They lose reputation when they accept proposals that are later reverted or fail calibration. This makes lazy rubber-stamp review costly without requiring reviewers to do more than they would already do well. The risk that staking selects against reviewers willing to engage hard syntheses is addressed by claim-difficulty-normalized review-credit (see [Reputation](#reputation)).

### Sub-topic creation

In v0:

- Proposed via `propose_sub_topic`.
- Curator evaluates against the criteria in [seed-topic.md](./seed-topic.md): articulable scope envelope, real disagreement, real audience, manageable corpus size, low political risk.
- The system computes feasibility hints from existing graph state: corpus density (do anchors exist?), anchor coverage (how much of the proposed scope envelope already has nodes?), projected closure distance.
- Curator accepts as `active`, defers as `proposed`, or rejects.

In Phase 3+: auto-discovery surfaces tractable scope envelopes from graph state; curator review remains as a check.

**Antenna nodes (idea, not committed).** A related Phase 3+ discovery mechanism: a node with no children — nothing derived from it yet — could carry "antennas," a small bounded set of embedded *hypothetical complements* describing what would be promising to cross with it. When a new real node lands close to an antenna in embedding space, the system auto-enqueues the candidate crossing as a frontier item. The value over plain nearest-neighbour lookup on the node itself is that a promising cross is often *complementary* rather than *similar* (an inhibitor of gene X is not semantically near a claim that gene X is upregulated), so the antenna encodes "what I'm looking for" rather than "what I am" — a HyDE-flavoured trick. The cost is the embedding-index fan-out, which is why the lean form would be an optional embedding on `open_question` nodes (already bounded by review) rather than materialised virtual claims, and why this stays an idea: not worth building until the graph is dense enough that ad-hoc lookup demonstrably misses these crossings.

## Identity

The identity model is a requirements sketch in v0; specific tech (OIDC providers, key formats, attestations) is a Phase 2 implementation choice — Phase 1 wired the gates and budget arithmetic against synthetic identities, but the production IdP and attestation issuance land with the public instance — and the *contract* the rest of the design depends on is fixed here.

Sybil-resistance has four layers: **binding cost** at issuance, **issuance-frequency cap** at the IdP, **per-identity rate-limit accounting** at the MCP write surface, and **cross-cause identity-clustering** at the curator surface. The first two live below the MCP layer — the server does not mint identities; the second two are server-side primitives. They compose multiplicatively against an adversary budget: a coalition affording K sybils pays per-sybil binding cost at mint, must spread the K mints across multiple epochs at the issuance-frequency cap, has each sybil's per-epoch throughput bounded by the rate-limit, and surfaces in the clustering projection if behavioral coordination survives the first three. The architecture's regression handle is the [sybil-amplified-coalition scenario](#adversary-testbed) where a fresh recruit walks past every *behavior-based* defense by construction (cluster signal, calibration record, reputation gates all require accumulated per-identity history); the four identity-layer primitives are what closes the seam the behavior-based stack cannot.

- **Bounded identities-per-real-person via binding cost.** Identity creation has a non-trivial cost — email verification at minimum, third-party OIDC (GitHub, ORCID, institutional SSO) preferred. The cost is tunable per IdP. Zero-cost identities are not supported. Each issued identity carries a numeric `attestation_level` recorded by the IdP at mint, opaque to the server (the server gates on threshold; it does not interpret what the level *means* — that's the IdP's job). The MCP-layer gate is `min_attestation_level`, consumed at every write tool that resolves a caller — `request_assignment`, all `propose_*` tools, and `cast_review_vote` — so the gate fires before any per-identity action accrues regardless of which tool the adversary chose. Refusal mode is `unauthorized` rather than the rep gates' `not_found` opacity, because the binding-cost mismatch is identity-level (a stable property of the IdP-issued credential) and there's no opaque-refusal value to protect: the adversary already knows their own attestation level. Read-path tools (`query_frontier`, `query_proposals`, `query_reputation`, `fetch_calibration_batch`) are not gated — read access doesn't multiply the per-identity action budget the gate is defending against. **GitHub is locked as the v1 production IdP** (slice 3c): the device-code OAuth flow mints identities through `GithubOAuthAuthenticator`, with attestation_level computed from the GitHub account signal — 2FA on + verified primary email + account age ≥ 30 days → level 2; anything weaker → level 1. Account-only OAuth without verified-email or 2FA caps at level 1 by construction. The mapping is intentionally conservative for v0 (a brand-new GitHub account with a verified email but no 2FA cannot reach level 2 on email-verification alone); the threshold days, the level-2 signal set, and future IdPs (ORCID, institutional SSO) are operationally tunable without changing the gate's contract — the server still only sees an opaque `attestation_level`. *Implementation status:* wired in the binding-cost slice — `attestation_level` is on the Identity contract, `min_attestation_level` is the testbed-tunable knob in `ReviewConfig`, the server-side gate fires at all 9 write-tool seams and refuses with `unauthorized`. The GitHub mapping is wired in slice 3c (`computeGithubAttestationLevel` in `packages/server/src/auth-github.ts`); the Identity contract carries `identity_provider` and the IdP-side `identity_provider_subject` so the same GitHub account resolves to a single Anchorage identity across re-signins (identity-on-first-signin). Default 0 leaves the gate inert and preserves existing-scenario behavior.
- **Issuance-frequency cap.** Per-(IdP, IP, ASN, …) per-epoch caps on the rate at which new identities can be minted. The cap lives below the MCP layer — the server does not mint identities and has no surface to enforce this directly. PRD commits the layer's existence; specific cap shapes are operationally private at the IdP. The cap is the *time* primitive on the cost-multiplier: an adversary affording K sybils still cannot mint them all in one epoch, which buys behavior-based defenses (cluster signal, calibration record, reputation gates) the accumulated history they need to fire. *Implementation status:* spec is settled here, the testbed-side fiction is wired as the `AdversaryBudget` primitive (see [Adversary testbed](#adversary-testbed)'s adversary-budget model) — the harness rejects mints that would exceed `issuance_cap_per_epoch` for the current epoch with a distinct `issuance_cap` failure mode — and the **IdP-layer cap is now wired in slice 3c** at `GithubOAuthAuthenticator.completeSignin` against the new `Store.idpIssuanceCounters` collection, bucketed per (provider, github_user_id) by default (per-account issuance throttling — the bucket-key dimension actually available at the seam). The new `issuance_cap` `ServerErrorCode` refuses signins past the cap before any identity mint or credential bind, so a refused signin burns no graph state. Per-(IP, ASN) bucketing requires request-side metadata that lands at the HTTP transport layer (slice 4) and composes into the same counter without a schema change; the *layer* and the failure-mode parity between sim and prod are what slice 3c commits to.
- **Per-identity rate-limit accounting.** Per-(identity, epoch) counters on observable write actions across the full write surface — `request_assignment`, all `propose_*` tools, and `cast_review_vote`. The cap is *single-bucket* across all write tools (not per-tool), so the cap measures total throughput per identity per epoch directly. When the counter hits the cap the tool refuses with a new `rate_limited` mode parallel to `not_found` and `unauthorized` — the contributor sees throttling with the recovery path "wait for the next epoch," not "obtain a higher attestation" or "no work available." The cap is the per-sybil-throughput axis on the cost-multiplier: at K sybils with per-sybil cap T, the coalition's per-epoch action budget is bounded by K × T. The signal is *global per identity* — the same identity's actions across two causes both consume the same per-epoch budget — for the same reason cross-cause anti-abuse is asymmetric to per-cause reputation: a sybil farm working two causes is more detectable than one working one, and the cost of opacity is small relative to the defense. The counter is consumed *after* `resolveCaller` and the binding-cost gate but *before* tool-specific business logic, so unauthorized callers don't burn budget while valid-input-but-business-logic-failed calls do (an adversary can't probe the system with bogus business inputs without paying for each). *Implementation status:* wired in the rate-limit slice — `rate_limit_actions_per_epoch` and `rate_limit_epoch_seconds` are testbed-tunable knobs in `ReviewConfig`, the `accountWriteAction` helper fires at all 9 write-tool seams (read-path tools are not gated, matching the PRD scope), and the new `rate_limited` ServerErrorCode is parallel to `unauthorized` and `not_found`. Defaults are both Infinity, leaving the gate inert and preserving existing-scenario behavior.
- **Cross-cause identity-clustering projection.** Curator-side projection (`server.curator.identityClusters({ window_seconds, min_signal })`) surfacing identity pairs whose behavioral fingerprint across causes suggests sybil coordination. The v0 signal is per-(reviewer pair) count of distinct causes where both reviewers cast votes on the same proposal — honest reviewers typically work in one cause (per-cause reputation; PRD §Reputation), so a pair appearing on shared proposals across multiple causes is the cross-cause behavioral fingerprint a sybil farm working multiple causes lights up and a single-cause coalition does not. Two metrics surface per pair: `cross_cause_count` (the headline signal) and `shared_proposal_count` (visibility into whether the pair coordinates across one shared proposal per cause or many — different shapes the curator weighs differently). The signal is *global per identity*, with documented governance and audit; specific signal shapes and thresholds are operationally private (methodology is public, tuning is not). Co-creation-timestamp correlation and vote-time correlation across causes are follow-up refinements on the same surface — they broaden the behavioral-fingerprint notion without changing the projection shape. *Implementation status:* wired in the clustering slice — `server.curator.identityClusters({ window_seconds, min_signal })` returns sorted (cross_cause_count desc, shared_proposal_count desc, ids asc) cluster records. Default `min_signal` is 2 (cross-cause coordination requires at least 2 causes to be a cross-cause signal); `window_seconds` is the rolling window over vote `created_at`, undefined for all-time.
- **Agents act as a human's delegate.** The human is the identity holder; agents are credentialed clients authorized to act on their behalf. Reputation, calibration outcomes, and accountability all attach to the human, not the agent. A human may bind several agents (a desktop MCP client, a long-running daemon, custom tooling) under the same identity; they all draw from the same per-(cause, sub-topic) reputation pool and the same single assignment slot per cause. Agent credentials are individually revocable by the human — losing a laptop or retiring a daemon does not require revoking the underlying identity — and curator revocation acts on the human, transitively disabling their agents. From the system's perspective an agent contributing on idle time and a human contributing directly are indistinguishable: both are tool calls authenticated as the same identity, drawing from the same assignment slot, accruing reputation to the same account. The bounded-identities-per-real-person property is preserved by construction; the agent layer does not multiply identities.
- **Pseudonymity is supported; anonymity is not.** A contributor may operate under a stable pseudonym; the system retains a binding between the pseudonym and the underlying identity-establishing credentials (email, OIDC subject) that curators can use under documented escalation. The graph and the public surface show the pseudonym; the binding is private.
- **Named credit on manuscript projections is opt-in.** Pseudonymous credit is allowed, but the project's recommendation is real-name credit for high-impact projections to retain academic legibility. Pseudonymous co-authorship is unusual and contributors should make that choice deliberately.
- **Revocation.** Identities can be revoked (sybil farms, terms-of-service violations). Revocation invalidates future participation without rewriting graph history; revoked contributions remain in the graph with the revocation flagged.
- **Authenticator seam.** The server-side trust boundary at the transport layer is a single `Authenticator.authenticate(token) → Caller` surface. Every MCP connection presents an opaque bearer token; the authenticator resolves it once at connection setup (or per-request, for the HTTP transport) into a `Caller` posture — the identity id, and the agent credential id when delegated. Downstream gates (binding-cost, rate-limit, reputation, assignment ownership) see only a resolved `Caller`; they do not interpret tokens and they do not branch on sim-vs-prod. Token grammar is the authenticator's private concern: the production runtime instantiates `GithubOAuthAuthenticator` (slice 3c) whose tokens are OAuth-issued session secrets resolved by hash lookup against `Store.agentCredentialSecrets` — the *same* index `HarnessAuthenticator` resolves bearer secrets against, so the seam's lookup shape is authenticator-agnostic; the testbed instantiates `HarnessAuthenticator` over the in-memory store with no network. Refusal at the seam is `unauthorized` — same code, same shape as the binding-cost gate downstream — so malformed, unknown, or revoked tokens fail before any per-tool gate accounts budget. This is the architectural primitive that lets PRD's "specific IdP tech is a Phase 2 choice" land without leaking implementation choice into the rest of the design: every gate downstream of the seam composes against the same `Caller` regardless of which authenticator produced it. *Implementation status:* wired in slice 3a (the seam) and slice 3c (the production concrete) — `Authenticator` and `HarnessAuthenticator` live in `packages/server/src/auth.ts`; `GithubOAuthAuthenticator`, `GithubApi`, the device-code orchestration, the attestation mapping, and `FakeGithubApi` (for tests and the testbed) live in `packages/server/src/auth-github.ts`. `Server` exposes `server.authenticator` (default `HarnessAuthenticator` over the Server's Store; injectable via `ServerDeps` at construction or via `server.setAuthenticator` post-construction for authenticators that depend on the Server itself), `buildMcpServer` accepts `{ token }` and resolves through it at build. Sim and prod use the same downstream surface by construction; the seam is exercised in CI through `mcp.test.ts`'s `authenticator seam` suite (six cases against `HarnessAuthenticator`'s bearer-secret path) and `auth-github.test.ts` (seventeen cases pinning the device-code flow, identity-on-first-signin, attestation mapping, issuance-cap, idempotency under retry, and refusal modes).
- **Agent-credential bearer tokens.** An `AgentCredential` is the issuance unit for transport-layer tokens (PRD §Identity, agent-as-delegate). At bind time `bootstrap.bindAgentCredential` mints a 256-bit URL-safe-base64 secret, returns it to the caller exactly once, and stores only its SHA-256 hash on the credential (`secret_hash`) and an inverted index entry mapping `secret_hash → agent_credential_id` (`Store.agentCredentialSecrets`). The plain secret never appears in any persisted record, server log, or audit trail; revocation flips the credential to `status: 'revoked'` without rewriting graph history. The production OAuth path uses exactly this surface: slice 3c's `GithubOAuthAuthenticator.completeSignin` calls the same `bootstrap.bindAgentCredential` that the testbed calls, and the secret it returns has identical wire-level shape (32-byte URL-safe-base64) to a testbed-issued credential's secret at the seam. The secret-source is `IdGen.bearerSecret()` so production gets `crypto.randomBytes`-driven entropy and tests/cassettes get deterministic counter-derived values — same pattern as id generation, no `if (sim)` branching. *Implementation status:* wired in slice 3b — `AgentCredential` gains the `secret_hash` field, `bindAgentCredential` returns `{ credential, secret }`, `Store.agentCredentialSecrets` is the new index across `MemoryStore` and `SqliteStore` (the slice-2 parity test exercises both backends through the bearer-secret path). `HarnessAuthenticator` accepts the bearer-secret grammar as the primary path; a transitional fallback to direct-identity-id tokens remains so the checked-in golden cassettes (`golden-deep-loop*.json`) replay without re-recording — both fallbacks resolve through the same `Authenticator.authenticate` surface, and the fallback retires when those cassettes are re-recorded.
- **GitHub OAuth (v1 production IdP).** Slice 3c locks GitHub as the v1 IdP behind the Authenticator seam. The device-code flow is the natural fit for desktop MCP clients (Claude Desktop, the Anchorage CLI, custom agents): the user authorizes once in a browser, the client polls for the access token, no client secret needs to live on the user's machine. `GithubOAuthAuthenticator.startSignin` returns `{ device_code, user_code, verification_uri, interval_seconds, expires_in_seconds }`; `completeSignin(device_code)` polls GitHub via the injected `GithubApi` seam (`GithubApiHttp` in production, `FakeGithubApi` in tests/testbed) and, on authorization, resolves the GitHub user id through `Store.identityProviderSubjects` — first signin mints a fresh Anchorage identity at the computed attestation_level and writes the (provider, subject) → identity_id index entry; subsequent signins by the same GitHub account re-use the existing identity (identity-on-first-signin; the bounded-identities-per-real-person invariant in code). Each signin mints a *fresh* agent credential under the identity, so losing a laptop and re-signing in produces a new bearer secret without colliding with the old one (which the human revokes separately). `completeSignin` is idempotent under retry: a client that lost the previous network response gets the same `(credential_id, secret)` back instead of a duplicate credential. The (provider, subject) index is the IdP-side hook the future ORCID/institutional-SSO authenticators plug into without changing the contract — the Identity record's `identity_provider` + `identity_provider_subject` fields are the wire-level dimension; new authenticators add a new enum value and reuse the rest of the surface. *Implementation status:* wired in slice 3c — `GithubOAuthAuthenticator`, `GithubApiHttp`, `FakeGithubApi`, and `computeGithubAttestationLevel` live in `packages/server/src/auth-github.ts`; `Identity.identity_provider` and `Identity.identity_provider_subject` land on the contract; `Store.identityProviderSubjects` and `Store.idpIssuanceCounters` are new collections across `MemoryStore` and `SqliteStore`; the slice-2 parity test exercises both backends through an IdP-driven mint. Production wiring decomposes across slice 4 (all three sub-slices closed 2026-05-14): the HTTP transport plus the device-code endpoints land in slice 4a; the OAuth App registration against `GithubApiHttp` and the production runtime entrypoint (`runProdServer` over env-configured `SqliteStore` + `LiveFetchVerifier` + `GithubApiHttp`, with the operator-facing guide at [docs/deploy.md](./deploy.md)) land in slice 4c.
- **HTTP transport for the Authenticator seam.** Slice 4a stands the seam up over HTTP. `packages/server/src/http.ts` exposes three surfaces on one Node `http.Server`: `POST /mcp` (plus the optional `GET`/`DELETE` the streamable-HTTP transport spec admits for SSE and session teardown) routes through the MCP SDK's `StreamableHTTPServerTransport` in stateless mode — the bearer token from the `Authorization: Bearer` header is resolved through `server.authenticator` once per request, a fresh `buildMcpServer` is bound to the resolved `Caller`, the transport handles the single JSON-RPC round-trip, and both are torn down on response close (the per-connection bind the in-memory transport uses, now per-request — the only thing that changed is the transport, the gate stack downstream is unchanged); `POST /auth/github/start` and `POST /auth/github/complete` expose the device-code flow unauthenticated, mirroring `GithubOAuthAuthenticator.startSignin` and `completeSignin` (these are how a brand-new client *acquires* its bearer secret in the first place); `GET /healthz` is the load-balancer liveness probe. Refusal mapping carries the typed `ServerErrorCode` through to HTTP status (`unauthorized` → 401, `rate_limited` / `issuance_cap` → 429, `invalid_input` / `invalid_state` → 400, `not_found` → 404) so a client branching on `code` reads the same payload it sees on the MCP wire. The slice's load-bearing assertion is that a bearer secret produced through the OAuth path (`POST /auth/github/complete` → `secret`) authenticates against `POST /mcp` identically to a secret produced by an in-process `bindAgentCredential` — sim and prod share the same Authenticator seam, the HTTP transport is just a transport. *Implementation status:* wired in slice 4a — `buildHttpHandler` and `startHttpServer` live in `packages/server/src/http.ts`; the seam is pinned in CI by `http.test.ts` (thirteen cases covering `/healthz`, the three `/mcp` refusal modes — missing-bearer, unknown-bearer, revoked-credential — an end-to-end MCP tool round-trip through `StreamableHTTPClientTransport`, the device-code → bearer-secret → `/mcp` sim≡prod round-trip, malformed-body refusals at `/auth/github/complete`, the issuance-cap → 429 mapping, the harness-only deployment posture where `/auth/github/*` is absent, and the 404/405 routing edges). TLS termination, edge rate-limit, sticky disk for `SqliteStore`, and backup live in the deployment surface ([docs/deploy.md](./deploy.md), wired in slice 4c — the runtime is intentionally edge-agnostic and the operator picks the concrete TLS terminator).
- **MCP-spec OAuth authorization server (client-driven signin).** The device-code flow above is usable but forces the human to be the protocol glue: hand-run the start/complete calls, copy a bearer secret, paste it into client config. The MCP authorization profile (OAuth 2.1, RFC 9728 + RFC 8414 + PKCE) exists precisely so an MCP client self-drives this. Anchorage implements it as a co-hosted authorization server (`packages/server/src/oauth.ts`, `OAuthProvider`): an unauthenticated `/mcp` answers `401` with `WWW-Authenticate: Bearer resource_metadata=…`; `/.well-known/oauth-protected-resource` (RFC 9728) names the AS; `/.well-known/oauth-authorization-server` (RFC 8414) advertises `authorization_endpoint`, `token_endpoint`, `registration_endpoint`, and — load-bearing, clients refuse a server without it — `code_challenge_methods_supported: ["S256"]`. GitHub is *not* a spec-compliant MCP AS (no discoverable RFC 8414 metadata, no PKCE discovery, no `resource` parameter), so the AS is co-hosted and **bridges to GitHub's browser authorization-code flow behind `/authorize`**. The access token the AS issues *is* the existing agent-credential bearer secret (`completeWebSignin` funnels through the same `mintFromAccessToken` tail the device flow uses — identity-on-first-signin, attestation, issuance cap, `bindAgentCredential` — so the two front doors cannot mint divergent identities), which means nothing downstream of `/mcp` changes: the same `extractBearer` → `authenticator.authenticate` → SHA-256 lookup runs byte-for-byte. The OAuth layer is a standards-compliant *front door* producing the same artifact the manual flow produced. Client onboarding becomes: point an MCP client at `/mcp`, authorize once in GitHub, done. **Audience binding (RFC 8707)** is satisfied structurally rather than by a JWT `aud` claim: a single-resource co-hosted AS that only ever mints tokens usable at its own `/mcp`, with the `resource` parameter validated equal to the canonical MCP URI at both `/authorize` and `/token`. **Confused-deputy mitigation** (spec §Confused Deputy, mandatory for a static-client-id proxy): `/authorize` renders a per-client consent interstitial naming the requesting client and its redirect host before the human is forwarded to GitHub. Client registration is RFC 7591 dynamic client registration (public clients, `token_endpoint_auth_method: none`, PKCE-protected — no client secret issued); Client ID Metadata Documents are a documented future addition. The registration redirect-URI policy follows RFC 8252 (OAuth 2.0 for Native Apps): https (claimed domains), http to a loopback address, and private-use URI schemes (`cursor://…`, `vscode://…`, `com.example.app:/cb`) are all accepted — installed clients that cannot host a public https endpoint register a custom-scheme callback, and several (Cursor among them) register a loopback and a private-use URI in the same batch, so the all-or-nothing batch check must accept the custom scheme or the whole registration fails. The security weight sits on mandatory PKCE plus the exact-match + open-redirect guard at `/authorize`, not on the scheme; a short denylist keeps browser-dangerous schemes (`javascript:`, `data:`, `file:`, `vbscript:`) out of the registry. The web bridge does change one operational posture relative to the device flow: GitHub's authorization-code token exchange is client-secret-authenticated, so the production deployment now holds a GitHub OAuth App **client secret** in the runtime environment (the device flow's no-secret property was a *consequence of that flow*, not a standalone security axiom — the secret is low-severity: `read:user user:email` read-only scopes, PKCE + exact redirect-URI + state mitigations — see [docs/deploy.md](./deploy.md)). AS state (registered clients, pending authorize sessions, ≤60s one-time codes) is in-process — the same single-instance posture and justification as the device-flow `flows` map (PRD §What's deliberately not specified here, scale-out). The whole surface is **additive and gated on the GitHub authenticator being wired** — exactly like `/auth/github/*` — so the testbed/`HarnessAuthenticator` posture never constructs it and is byte-for-byte unchanged (the sim≡prod invariant: no `if (sim)` branch, the OAuth front door simply does not exist without prod IdP config). *Implementation status:* wired in the MCP-OAuth slice — `OAuthProvider` plus the discovery/registration/authorize/callback/token routes in `packages/server/src/http.ts` (all gated on `opts.oauth`), the `GithubApi` seam gains `webAuthorizeUrl` / `exchangeWebCode` (real impl client-secret-authenticated; `FakeGithubApi` scripts both), and `finalizeAuthorization`'s tail is extracted to the shared `mintFromAccessToken`. Pinned in CI by `oauth.test.ts` (metadata shapes, DCR redirect-URI policy, `/authorize` validation, the full register → authorize → GitHub-callback → token bridge yielding a token that authenticates at the seam, and the token failure modes — PKCE mismatch, replay, expiry, redirect/resource mismatch, wrong grant). Production env wiring (`ANCHORAGE_GITHUB_CLIENT_SECRET`, `ANCHORAGE_PUBLIC_BASE_URL`) and the operator walkthrough land in [docs/deploy.md](./deploy.md).
- **Roles (curator vs contributor).** Identities carry a `role: 'contributor' | 'curator'` field. `contributor` is the default — every IdP-driven signin (slice 3c GitHub OAuth, future ORCID, future institutional SSO) lands here, and the contributor-facing write surface (`request_assignment`, `propose_*`, `cast_review_vote`, etc.) is the only one a contributor exercises. `curator` is the privileged role per PRD §The contribution flow (Resolve step) and §Reviewer assignment (step 4: curator escalation): the human-or-humans who can fire `server.curator.*` (`acceptProposal`, `rejectProposal`, `deferSubTopic`, `revokeIdentity`). The role is invariant under IdP signin — only `'harness'`-provider mints (i.e. the admin CLI) can produce a curator, so no GitHub account can auto-promote into the curator pool. Curator seating is an operator decision; `bootstrap.mintIdentity` refuses `{ identity_provider: 'github', role: 'curator' }` with `invalid_input` at the server seam. *Implementation status:* wired in slice 4b — `IdentityRole` lands on the contract with `Identity.role` defaulting to `'contributor'` (so pre-4b stored records and cassettes parse cleanly through the default without re-recording); `MintIdentityInput.role` is optional and consumed by `bootstrap.mintIdentity`, which enforces the role-provider invariant. ## Reputation

Reputation is structured to resolve a real trilemma the design cannot wave away: *slow* decay rewards consistency (the design goal) but lets patient adversaries stockpile; *fast* decay neutralizes stockpiles but disenfranchises episodic experts (the part-time clinician is exactly the contributor we want); *review-as-staking* punishes lazy review but selects against accepting hard syntheses (which are riskier to stand behind). Acknowledging this directly:

- **Two-component reputation.** A *demonstrated-competence* component, slow-decay, gates eligibility tiers (who is in the reviewer pool at all) — `assignment_min_demonstrated`, consumed at `request_assignment` with the opposite null-policy from the recent gate (callers with no rep entries in the cause fail by construction when the threshold is positive — the demonstrated tier is "have you proven yourself yet?", and an unproven identity is by construction not in the pool). A *recent-activity* component, fast-decay, gates assignment (who is drawn for a given proposal) — `assignment_min_recent`, consumed at the same seam, with fresh-reviewer bypass so first-assignment bootstrap stays open. Both gates refuse with the same `not_found` mode, structurally indistinguishable from "no work available." A patient adversary can stockpile competence but must remain currently active to be assigned — and visible activity is detectable.
- **Per-(cause, sub-topic), accrued from assigned work.** Anchored at the cause level (the unit of belonging), refined by which sub-topics a contributor's *assigned* work has actually landed in. Sub-topic rep is therefore an emergent record of where the system has routed someone, not a self-declared specialty. This closes both the rep-laundering vector (no easy-sub-topic shopping) and the coalition vector (contributors can't pre-arrange to land on each other's proposals). Contributor-initiated work earns sub-topic rep at a substantially reduced weight, conditional on independent confirmation, to preserve a genuine novel-synthesis path without making it the laundering route.
- **Earned through confirmed contributions and accurate reviews.** Both contributing nodes that survive *and* reviewing accurately count.
- **Lost through reverted contributions and inaccurate reviews.** Supersedes and rejected calibration items both decrease reputation. Self-supersedes (a contributor superseding their own node) do not count toward survivorship — only supersedes by other contributors do.
- **Review-credit normalized by claim difficulty.** Without normalization, the regime selects for reviewers who accept easy proposals. Difficulty proxies — review effort, prior divergence, sub-topic frontier-distance — weight review-credit so that engaging hard syntheses is not dominated by rubber-stamping easy ones. The first-iteration difficulty proxy is the same per-proposal contention signal the cluster-stratification refinement already computes (`2 * min(accepts, rejects) / total_votes`): unanimous-easy items contribute zero contention to the cluster signal and earn reduced rep credit by the same primitive; perfectly-split contentious items contribute the full signal both places. The wired knob is `review_credit_contention_alpha`: reviewer rep deltas (gain on accurate, loss on inaccurate) scale by `alpha + (1 - alpha) * contention`, where `alpha = 1` (default) recovers the v0 uniform-credit regime exactly and `alpha < 1` is the difficulty-aware regime. *Implementation status:* the alpha primitive is wired and pinned end-to-end at three cells (alpha=0 unanimous → zero reviewer credit, alpha=0 split → contention-scaled credit, alpha=0.5 unanimous → half-credit). Default 1.0 preserves the v0 behavior; calibration credit (PRD §Calibration batches) flows through `bumpReputation` directly and is *not* alpha-scaled — the calibration path is ground-truth-individual, not convergence-derived, and stays a pure-honesty channel for the patient archetype's accumulation. The fifth parameter sweep cube re-baselines the patient-adversary gate-threshold cube under alpha < 1: with cube #2's thresholds (recent=0.5, demo=1.5) calibrated against the alpha=1 bootstrap demonstrated of 2.0, alpha=0.5 cuts the bootstrap demonstrated to 1.0 and the demo gate's value-comparison threshold (1.5) false-positive-locks-out honest reviewers at first request_assignment — the patient archetype's drift never starts, but the closure observation is misleading because honest review collapsed at the same gate. The cube reads ASR alongside a separate lockout-rate metric so "attack closed by gate" is distinguishable from "honest pool collapsed at the same threshold," and pins re-tuned thresholds at alpha=0.5 (recent=0.5, demo=0.75) that recover cube-#2's closure without the false-positive — demo scales by alpha to preserve honest-pool headroom; recent scales less because its quiet-window decay is dominated by alpha-invariant calibration credit. The patient-adversary signature (priming through easy unanimous accepts on well-grounded excerpts to harvest rep cheaply) is exactly the pattern alpha < 1 makes expensive on the convergence path; the calibration path stays the same drift-bandwidth seam it was at alpha=1. Proposer rep deltas are unscaled (the PRD passage scopes difficulty-normalization to *review* credit; the proposer's accept/reject outcome carries its own weight independent of how much disagreement the reviewers had on the way there).
- **Eligibility tiers public; numeric reputation private.** Contributors can see what tier they are in (and what gates the next tier); raw numbers are not leaderboards. Reviewers receive batch-level performance feedback after-the-fact, not in real time. *Implementation status:* the v0 tier mapping is wired in slice 5c — three tiers per (cause, sub-topic), derived from the contributor's decayed `(demonstrated, recent)` against the same `assignment_min_demonstrated` and `assignment_min_recent` review-config thresholds the gate at `request_assignment` already consumes (PRD §Reputation, Two-component reputation, above). The tiers are: **none** (no entry for this (cause, sub-topic), OR `demonstrated < assignment_min_demonstrated` — by construction not in the pool, matching the demonstrated gate's opposite null-policy); **quiet** (`demonstrated >= threshold` but `recent < assignment_min_recent` — proven contributor in a between-windows state, the episodic-expert case in dormant phase); **contributing** (both gates pass — would not be filtered at `request_assignment` for this (cause, sub-topic)). The public projection lives on the `contributor://{id}` MCP resource (`@anchorage/contracts` `ContributorProfile`, `PublicReputation`, `PublicReputationTier`): a deliberately narrow `PublicContributor` (id, display_name, created_at, status — *not* `identity_provider`, `identity_provider_subject`, `attestation_level`, or `role`) plus an array of `PublicReputationEntry` carrying tier labels only. Raw `demonstrated`/`recent` values stay reserved for the contributor's own `query_reputation` tool, which returns numbers to the caller's own caller (the contributor needs them to reason about their own pool position). When the operator runs gates inert (`assignment_min_*` = 0, the default), every entry collapses to `contributing` and the projection answers "has any rep here, yes or no" — tier richness scales with the operator's chosen thresholds, by construction, with no separate tier-threshold knob. The mapping uses the *entry's own* components rather than the per-cause maximum the assignment gate computes across sub-topics, so the tier reflects "this contributor's track record in *this specific sub-topic*" — matching PRD §Reputation's anchored-at-cause-refined-by-sub-topic framing.
- **Non-transferable, non-monetizable.** Reputation is a coordination signal, not a token.
- **Specific formulas tuned in testbed.** Initial values are chosen for testbed simulation; production values are confirmed against attack-success-rate measurements.

## Credit

When a sub-topic produces a manuscript projection, contributor credit is computable from graph state.

The basic shape of the credit function:

- **Node provenance.** For each node included in the projection, who proposed it, who reviewed it, when.
- **Survivorship weighting.** Nodes that survived multiple supersedes events count more than nodes that barely survived one review.
- **Load-bearing weighting.** Nodes whose removal would break a projection chain count more than peripheral nodes.
- **Review weighting.** Reviewers who voted correctly accrue partial credit, weighted lower than proposers.

Credit is **deterministic given graph state and projection config**. It is not negotiated in a meeting. Authorship disputes route to (a) a versioned change to the projection config, evaluated by the same governance-change CI as any other governance change, or (b) a governance change to the credit function itself, also CI-evaluated. Both paths are public and auditable. Projection configs are not a back-channel.

Specific weights are deferred to the testbed. The v0 credit function (wired by [Phase 2 slice 6a](../ROADMAP.md#phase-2--single-cause-public-instance-opened-2026-05-14) inside `server.resources.getManuscript`) composes the four factors as `units_per_contribution = base_weight * (1 + survivor_bonus * supersedes_predecessors + load_bonus * induced_derives_degree)`, with proposer credit at `credit_proposer_weight` and converged-accept-aligned reviewer credit at the strictly smaller `credit_reviewer_weight`. All four knobs (`credit_proposer_weight`, `credit_reviewer_weight`, `credit_survivor_bonus_per_supersede`, `credit_load_bonus_per_induced_derives`) live on `ReviewConfig` alongside the other testbed-tunable thresholds, with non-zero v0 defaults (no baseline to preserve — the projection is new); the testbed will sweep them once a Phase-2 corpus exists to read deltas off of. Revoked-status contributors remain in the credit list with the status flagged (PRD §Identity: "past contributions remain in the graph with the revocation flagged").

## Adversary testbed

The testbed is the project's continuous-integration story for governance — and the cheapest credible artifact Anchorage can ship.

### Architecture

- **Real graph schema.** The testbed uses the same schema, the same write-path tools, the same MCP server interface as the public instance.
- **Simulated contributor population.** Synthetic contributors connect via the same MCP protocol, salted with the adversary taxonomy below.
- **Two engine tiers under one population.** The fast-loop archetypes are *scripted*: deterministic content providers, review deciders, and fabricators that stand in for "a model on the other end of the connection" at zero API cost and with replayable behavior — this is the bulk of CI. The deep-loop archetype is *model-backed*: `runLlmAgent` is a generic MCP-tool agent loop — list the server's tools, hand their JSON schemas to the model's tool-use API, execute each `tool_use` block against the MCP client, feed the result back, repeat until the model stops or a turn budget is hit. It carries no Anchorage-specific logic; the role (honest contributor, strategic adversary, patient adversary, lazy reviewer) lives entirely in the system prompt, so the agent sees exactly the surface a real client sees and can do exactly what a real agent could. *Implementation status:* `runLlmAgent` is wired in the testbed package over the Anthropic Messages API (no SDK dependency — the loop speaks the API shape directly); a `pnpm --filter @anchorage/server live` runner points it at the wired in-process server for the manual "is the loop real" check, and an integration test drives the same loop with a scripted model so the path is CI-exercised without a key. The first two named roles — honest-strong, patient adversary — are wired as `LlmRole` configs (system prompt plus cause-keyed task message) in the testbed package and CI-exercised through the scripted-model path, so the role prompts are pinned and reviewed alongside the loop rather than written ad hoc at each call site (see §Continuous integration's implementation note). Surfacing the error-shape constraint a tool-listing client imposes — an MCP client that has called `tools/list` validates any `structuredContent` against the tool's output schema, so error results carry their typed `{ code, message }` payload as JSON text in `content` rather than as `structuredContent` — was part of wiring this; the typed code still survives the transport unchanged.
- **Adversary budget model.** Each adversary archetype carries a numeric **budget** the harness deducts from as the archetype acts. Minting an identity at attestation level α costs α (the [binding-cost](#identity) layer); minting under the issuance-frequency cap deducts at most N identities per epoch (the [issuance-frequency-cap](#identity) layer); each per-epoch observable write action consumes one unit of that identity's per-epoch capacity at the configured rate-limit T (the [rate-limit-accounting](#identity) layer); the [clustering projection](#identity) costs the adversary nothing directly but surfaces them at the curator surface. ASR is measured at fixed budget across the defense-threshold sweep; the sweep dimension PRD §Identity names — coalition-affordable-identities-per-epoch — is the budget axis, and parameter sweeps measure ASR as a function of budget at each defense configuration. The model is the harness-side fiction the testbed needs because issuance is a layer below MCP (the server sees only the issued identity record, not the cost paid for it), but the budget arithmetic is the testbed's regression handle on whether the four identity layers compose to make a target adversary infeasible. *Implementation status:* wired in the budget-model slice as the `AdversaryBudget` primitive — `tryMint(epoch)` operationalizes the binding-cost (α deducted from B) and issuance-cap (≤ N mints per epoch, distinct `budget` and `issuance_cap` failure modes) layers; `tryAct(identity, epoch)` operationalizes the per-(identity, epoch) throughput cap T mirroring the server-side `accountWriteAction` arithmetic. The four-layer architecture's full composition is now CI-checked across all the named regression-handle attack patterns.

### Why assignment-driven contribution closes several attack surfaces

Several adversary vectors that previously needed elaborate defenses collapse under assignment-driven contribution:

- **Reputation laundering** (build rep on easy sub-topics, parlay into contested ones) — closed by construction. Capacity is cause-level; sub-topic is the system's choice; sub-topic rep is an emergent record of routing.
- **Coalition pre-arrangement** (multiple identities landing on each other's proposals) — closed by construction. Reviewer assignment is system-driven; coalitions cannot pre-arrange the queue.
- **Calibration distinguishability via batch-level signatures** — narrowed substantially. Calibration items already arrive via assignment; with proposals also arriving via assignment, calibration is structurally indistinguishable from "the rest of the work" — it is just another assigned task.
- **Cherry-picking proposals to review** — closed. Reviewers do not select proposals; the system assigns them.

What remains, and what the testbed continues to exercise:

- **Patient adversary accepting clean assignments** to build rep over time, then drifting on a contested assigned proposal. Two-component reputation with fast-decay recent gating assignment (`assignment_min_recent`) + difficulty-aware review credit (`review_credit_contention_alpha`) are the wired convergence-path defenses; calibration injection is a parallel ground-truth-individual channel that stays alpha-invariant by construction.
- **Sybil farms providing capacity to absorb assignments** — addressed by the four-layer sybil-resistance architecture in [Identity](#identity): binding cost via `min_attestation_level` at the MCP write surface, issuance-frequency cap below the MCP layer, per-identity rate-limit accounting via the `rate_limited` refusal mode, and cross-cause identity-clustering at the curator surface. The four layers compose multiplicatively against the adversary's budget under the [budget model](#architecture) above. *Implementation status:* all four layers plus the testbed-side adversary-budget model are wired (ROADMAP §Status), and cubes #4-#7 CI-check the multiplicative composition across the named regression-handle attack patterns. The standalone sybil-amplified-coalition scenario in the [adversary taxonomy](#adversary-taxonomy) stays as the regression handle on the freshness-bypass observation in isolation; the MCP-layer closure stack composes with the identity layer below.
- **Hallucinated submissions on assigned tasks** — verification engine + reviewer responsibility.

### Adversary taxonomy

The testbed exercises the regime against the following synthetic contributor populations:

- **Honest-weak.** Modest-capability honest contributor (e.g. small local model). Should largely succeed; failure-to-contribute rate measures friction.
- **Honest-strong.** Frontier-model honest contributor on hard synthesis tasks. Should succeed even on cases where simpler contributors stall.
- **Lazy.** Reviewer that votes without reading. Should be caught by calibration.
- **Hallucinator.** High-temperature model with no grounding discipline. Should be caught at the verification engine (span mismatch, unresolved citations) before review.
- **Strategic adversary.** Hidden-objective model — instructed to bias the graph toward outcome X while passing calibration. Tests whether redundant peer review and reputation gates close the attack surface.
- **Patient adversary.** Strategic adversary with a long horizon — builds reputation honestly for months before drift attempts. Tests whether per-(cause, sub-topic) reputation tracking and review-as-staking prevent slow drift.
- **Sybil farm.** One operator running many identities at the per-identity cost set in the [identity model](#identity). The four-layer sybil-resistance architecture (binding cost, issuance-frequency cap, per-identity rate-limit accounting, cross-cause identity-clustering) composes multiplicatively against the operator's budget under the [adversary budget model](#architecture). *Implementation status:* all four layers plus the testbed-side adversary-budget model are wired (ROADMAP §Status), and cubes #4-#7 CI-check the multiplicative composition across the named regression-handle attack patterns — cube #7 specifically reads the freshness-bypass closure where the identity layer (binding cost) closes the seam the cluster signal cannot reach. The complementary standalone testbed-side regression handle is the sybil-amplified-coalition scenario where a fresh recruit walks past every *behavior-based* defense by construction (cluster signal, calibration record, and reputation gates all require accumulated per-identity history); the MCP-layer closure stack composes with the identity layer below.
- **Coalition.** Multiple identities coordinating reviews on each other's proposals. Tests whether reviewer-assignment stratification, calibration random-sampling, and the collusion-detection sample-size floor defeat coordination — and how the regime degrades on small sub-topics where the floor isn't reached.

### Continuous integration

Every governance change — calibration ratio, reputation formula, reviewer-assignment algorithm, sub-topic-creation rule, credit-function weight — runs against the adversary suite before merging. Proposals that fail simulation against known attacks do not merge.

The CI envelope has two tiers. A *fast loop* (cheap simulated populations, large numbers, low compute) runs on every governance-change PR. A *deep loop* (expensive populations including frontier-model adversaries, smaller numbers, higher compute) runs on a slower cadence and on changes that the fast loop flags as borderline. The honest framing is "a year of patient adversarial behavior in an hour" applies to the fast loop's cheap populations; the deep loop is where frontier-model patient adversaries live, and that loop is genuinely expensive. *Implementation status:* the deep-loop engine is the `runLlmAgent` archetype (the generic MCP-tool agent loop described under [Architecture](#architecture)), and the model-backed roles — honest-strong, strategic adversary, patient adversary — are wired as `LlmRole` configs in the testbed package: a system prompt carrying the role definition (PRD §Adversary taxonomy's language directly; the patient adversary's prompt makes the build-reputation-then-drift strategy explicit, the strategic adversary's the drift-now-on-any-contested-item one — both honest on routine work, neither ever risking a calibration misfire — and each takes the hidden objective as a parameter) plus a cause-keyed task message, with the agent loop role-blind by construction. The role prompts are the deep loop's experimental treatment, so they are pinned in one place and CI-exercised with a scripted model (the scripted model can't exhibit a role's distinguishing behavior — it drives the honest write-path that the patient adversary's build phase is by design indistinguishable from — but it proves each config plugs into the loop); the `live` runner picks a role from an env var so it and CI run the literal same configs against a real model. The deep-loop scenario that stands up a contested item — so the patient adversary's drift half is observable — is wired as `run-deep-loop.ts` over a shared seeded fixture (`deep-loop-scenario.ts`, the population-run analogue of `live-scenario.ts`), and a golden cassette of a small `ci`-preset run (2 honest-strong + the patient adversary, a 2-anchor excerpt frontier, a 2-item calibration corpus) replays it deterministically in CI (`golden-deep-loop.test.ts`) — the same regression-handle shape the single-agent `live` runner gets from `golden-cassette.test.ts`, so a drift in the agent loop, the tool surface, the population round loop, or the patient-adversary prompt surfaces as a cassette miss in CI. The model-backed roles are joined to the *parameter-sweep cube* layer that runs alongside the scripted populations as `run-deep-loop-cube.ts`: one model-backed cell per setting of a swept axis — four axes today, the calibration defense on versus off (against the patient adversary), the adversary role (a strategic-adversary cell on the brazen contested item, with `votes_to_reject` raised to 3 so the lone adversary, which runs after the two honest reviewers each round, is offered the contested review before the two honest rejects would otherwise close it), contested-item severity (a `borderline-contested` cell that holds the strategic-adversary role fixed and swaps the brazen overstatement for one the source's observational "associated with" framing is plausibly readable as supporting causal "increases", plus a `borderline-surrogate` cell that runs a second structurally different borderline drift — surrogate-substitution: source describes ctDNA clearance as a prognostic marker, claim restates it as a therapeutic target — and a `borderline-subgroup` cell that runs a third — generalization-from-subgroup: source reports an effect confined to MMR-deficient stage III colon cancer, claim drops the subgroup qualifier and generalizes across resected CRC — both under the same bare v0 closure stack), and closure-stack version (a `borderline-contested-v1` cell with `escalation_revise_counts_as_reject` ON, a `borderline-contested-v2` cell with the strict v1 stack — both escalation knobs ON, also requiring `accepts >= votes_to_accept` at escalation — and a `borderline-contested-v3` cell adding `contested_votes_to_accept` on top, the first closure-stack knob to touch the auto-close-accept path), with the cell list (`DEEP_LOOP_CUBE_CELLS` in `deep-loop-scenario.ts`) the extension point for more — each cell its own cassette, replayed deterministically in CI by `golden-deep-loop-cube.test.ts`. The calibration on/off pair is the model-backed complement to the scripted calibration-aware cube: that cube shows the calibration defense is load-bearing when an adversary actually drifts; this one checks that on the honest baseline — where the model adversary keeps to the build-standing-first strategy its prompt encodes — flipping the defense off moves no outcome, so it carries no false-positive cost. The strategic-adversary cell on the brazen item is the model-backed counterpart to the scripted strategic-adversary closure cubes: the recorded run has the strategic adversary reviewing the overstated claim and rejecting it — the "improves response rates" overstatement too brazen for its pick-the-borderline-cases strategy to defend — with redundant honest review carrying the closure regardless of how the adversary votes (a 3-0 reject, or a curator-resolved 2-1 had it drifted). The `borderline-contested` cell is the cube's *first recorded closure failure*, and the finding the cube was built to surface: handed the borderline item, the strategic adversary plausibly drifts (the recording: votes accept — the pick-the-borderline-cases strategy *will* defend an observational-vs-causal framing gap), one honest reviewer splits the rejecting side with a careful *revise* (the source's caveats *should* be reflected in the claim's wording, but revise is not a vote against acceptance in the v0 aggregation), and the curator escalation pass closes the 1 accept + 1 reject + 1 revise *toward accept* (its v0 rule accepts on a tie). The overstated claim ends `accepted` — the v0 closure stack (`votes_to_reject` threshold + revise not counting as a rejection + curator accept-on-tie) does not contain model-driven drift on a borderline contested item when a careful "revise" splits the rejecting vote. The v1 closure stack against that finding is a pair of orthogonal knobs on the curator-escalation tiebreak. (1) `ReviewConfig.escalation_revise_counts_as_reject` (default `false`) flips the escalation rule from `r > a ? reject : accept` (revise ignored, accept-on-tie) to `r + revise > a ? reject : accept` (revise counted as a soft-reject — capturing the load-bearing fact that a careful reviewer voting "revise" is not endorsing the proposal). (2) `ReviewConfig.escalation_requires_votes_to_accept` (default `false`) additionally requires `accepts >= votes_to_accept` for escalation-to-accept — the same affirmative-supermajority floor the auto-closure path enforces during normal voting — so a single accept vote, or a plain 1-1 tie with no revise, no longer escalates to accept on the accept-by-default rule. The two compose: the rule is `(reject_side > a) OR (knob2 AND a < votes_to_accept) ? reject : accept`, with `reject_side = knob1 ? r + revise : r`. Both knobs govern only the *escalation* tiebreak, not the auto-closure thresholds; the harness-level v0/v1 (1-1-1) and v0/v2 (1-1-0) deltas are pinned byte-for-byte by scripted-decider pairs in `population-loop.test.ts`. The `borderline-contested-v1` cube cell is the v1 stack's real-model regression baseline (sampling is non-deterministic, so its recording is a different rollout from the v0 cell's: 1 accept + 2 reject + 0 revise, curator-escalated reject under both rules — confirms v1 doesn't break the realistic borderline run). The `borderline-contested-v2` cube cell — the strict stack, both knobs on — recorded the cube's *second* closure failure, at a *different path* than the v1 knobs reach: in this rollout the strategic adversary voted accept on the contested item AND at least one honest reviewer also voted accept, hitting `votes_to_accept=2` and auto-closing the proposal accepted via the *normal vote path* before the curator escalation pass could see it. The v1 knobs govern only the escalation tiebreak; the auto-close-accept path is unfortified. The v3 closure-stack knob against that finding is `ReviewConfig.contested_votes_to_accept` (default `0`, inert): when > 0 and the proposal has any reject or revise vote in its tally, the auto-close-accept threshold is raised from `votes_to_accept` to this value — so a 2-accept-1-revise tally that would auto-close accept under v0 at `votes_to_accept=2` does not auto-close at `contested_votes_to_accept=3` because `acceptCount=2 < 3`; the proposal is held for the curator escalation pass, where the same contested floor applies as an additional constraint on escalation-to-accept (`accepts >= contested_votes_to_accept` required when dissent is present, otherwise escalates reject). Self-contained: composes with v1 (`escalation_revise_counts_as_reject`) and v2 (`escalation_requires_votes_to_accept`) but does not require them — v3 alone closes the auto-close-path failure the cube's `borderline-contested-v2` cell recorded by holding the 2-accept-1-revise tally on the auto-close side and rejecting it on the escalation side. v3 is the first closure-stack knob to touch the auto-close-accept path; v1 and v2 govern only the curator tiebreak. The load-bearing v0/v3 delta is pinned byte-for-byte at the harness level by the scripted-decider pair in `population-loop.test.ts`: same 3-reviewer accept/revise/accept sequential setup, knob inert versus `contested_votes_to_accept: 3` — the v0 case auto-closes every excerpt accept on the second accept vote (0 escalations, every excerpt materialized as a node); the v3 case holds every excerpt off the auto-close path and the between-rounds curator pass escalates each reject (0 excerpt nodes materialized, every proposal `rejected`). The model-backed `borderline-contested-v3` cube cell is the v3 closure stack's real-model regression baseline on the borderline ctDNA-MRD scenario: the recorded rollout happens to exercise v3 directly — strategic adversary drift accept + a confused honest accept + an honest reject (2-accept-1-reject-0-revise), which under the v2 stack would auto-close accept at `votes_to_accept=2` but under v3 is held off the auto-close path (`acceptCount=2 < contested_votes_to_accept=3` with dissent) and escalates reject (`a=2 < contested_votes_to_accept=3` at escalation). The contested overstated claim ends `rejected`. Two further contested-item-severity cells, `borderline-surrogate` and `borderline-subgroup`, then read whether the v0 closure failure the `borderline-contested` cell records generalizes across drift patterns. `DEEP_BORDERLINE_SURROGATE` is surrogate-substitution drift — source describes ctDNA clearance as a prognostic marker (clearers under standard adjuvant therapy have better RFS, a selection-into-better-prognosis observation), claim restates it as a therapeutic target (achieving clearance improves RFS — the *kind of relationship* the verb implies slips, not its modality). `DEEP_BORDERLINE_SUBGROUP` is generalization-from-subgroup drift — source reports an effect confined to MMR-deficient stage III colon cancer (a molecularly distinct subset with separately-documented immune behavior), claim drops the subgroup qualifier and generalizes across resected CRC (the *population scope* widens, verb modality and kind of relationship both preserved). Both cells run under the bare v0 closure stack, and both recordings close the drift cleanly via *different paths*: in `borderline-surrogate` the strategic adversary's vote-cast attempts hit assignment_id mismatch errors and the honest pool's 0-accept-1-reject-1-revise tally is escalated reject by the v0 curator pass; in `borderline-subgroup` all three contributors (both honest reviewers plus the strategic adversary electing strategic discipline) vote reject on the contested item, hitting `votes_to_reject=3` and auto-closing on the normal vote path within round 1 with 0 escalations. So the contested-item-severity axis now reads three drift kinds with three different closure paths: verb-swap drift breaks v0 (curator accept-on-tie); surrogate-substitution drift v0 closes (curator escalation reject on a 0-1-1 tally); generalization-from-subgroup drift v0 closes (auto-close-reject on a 3-0 tally). Two of three drift kinds: v0 holds. Two further sampling rollouts of the verb-swap cell — `borderline-contested-r2` and `borderline-contested-r3`, identical opts to `borderline-contested`, only the cassette path differs — then resolve whether the original v0 verb-swap break was a single sampling-unlucky rollout. r2 records v0 holding (adversary drifts but both honest reviewers reject cleanly, 1-2-0, curator escalation reject); r3 records a *second failure path* on v0 (adversary drift + a confused honest accept hits `votes_to_accept=2` and the proposal auto-closes accept via the normal vote path before escalation can see it — the same auto-close-path failure shape the `borderline-contested-v2` cell records under the strict v1 stack, which proves the auto-close failure path is reachable on the bare v0 stack and not a v1/v2 artifact). So the verb-swap-on-v0 trio is 2/3 fail via *two distinct paths* (escalation accept-on-tie at 1-1-1; auto-close-accept at >=2 accepts), 1/3 holds — strengthening the v3 closure stack's motivation: the v0 verb-swap break recorded on the original `borderline-contested` cell was not one sampling-unlucky path, it was one of multiple failure paths present on the scenario, both closed by v3. The axis now maps a drift-pattern landscape with sampling-variance reads on the failure cell rather than reporting a single drift's closure outcome. What remains in this thread is broadening the cube further (more borderline items varying the kind of overstatement, more closure-stack cells exploring orthogonal auto-close rules) and standing up the slower-cadence CI job that runs the model-backed loop on its own schedule.

Threshold definitions ("what counts as passing") are themselves versioned. Threshold changes go through the same CI process; threshold-loosening proposals require multi-curator approval and are publicly logged. The testbed protects against the simulated attacks; the threshold-versioning regime protects the testbed from being defeated by changing the meaning of "pass."

CI fails-shut: if the testbed infrastructure or model APIs are unavailable, governance changes do not merge. Liveness is sacrificed for safety. This is a deliberate choice and is acknowledged.

### What the testbed does *not* cover

- **Sophisticated human adversaries with out-of-band coordination.** State actors, well-funded interest groups, attackers willing to coordinate by phone, and attackers exerting political pressure on identifiable curators are not simulated. The residual risk is met by non-simulation defenses: curator vigilance, audit log, post-hoc forensics, and federation as an exit.
- **Senior expert hand-prose contributions.** Models do not reproduce these well. The simulation under-approximates the friction the regime imposes on them. Worth measuring against real contributors as soon as Phase 2 begins.
- **Phase 3+ failure modes.** Cross-cause reputation transfer, federation-induced collusion, multi-instance state synchronization. These are out-of-scope for Phase 1 testbed by design and gain testbed coverage when their phases open.
- **Novel attacks not yet in the adversary suite.** Passing the testbed is necessary, not sufficient. Continuous addition of new adversary archetypes is part of the testbed's maintenance cost.

This is the core load-bearing claim of the project, stated honestly. Wikipedia debates policy changes for years because it cannot test them; Anchorage tests them in an afternoon for the threat classes the testbed covers, and is honest about the threat classes it does not.

## What's deliberately not specified here

These are intentionally open until the testbed surfaces them:

- **Numeric calibration ratios** (what fraction of a reviewer's batch is calibration items).
- **Reputation formula constants** (gain weights for proposing, reviewing, surviving supersedes; decay rates).
- **Reviewer pool sizes** (N for a given proposal class).
- **Vote-aggregation thresholds** (what counts as convergent vs divergent).
- **Federation contracts** (Phase 3+ — when independent Anchorage instances exist, how their state and reviewer pools relate).

Specifying these before testing them would be guessing. The testbed exists exactly to replace guessing with measurement.
