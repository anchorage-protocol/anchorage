# Roadmap

Anchorage is built phase by phase, with each phase producing a discrete artifact that's useful on its own and informs the next. This file is honest about what's planned, what's hand-waved, and what's load-bearing for everything else.

The roadmap is a living document. Phases will move; some will split or merge. What stays stable is the *order of dependencies*: each phase produces something the next phase needs, and we don't skip ahead.

---

## Phase 0 — Design (current, concurrent with Phase 1)

**Goal:** A coherent design document set that the next person to read can decide *in ten minutes* whether they want to be involved.

**Concurrency.** Phase 0 and Phase 1 run concurrently. Doc work and testbed work co-evolve, with the docs-never-drift discipline (see [CLAUDE.md](./CLAUDE.md)) keeping them in lockstep — every commit that touches a contract updates both. Phase 0 finishes when the docs stop moving under their own pressure; testbed work continues into Phase 1's exit.

**Artifacts:**

- [README.md](./README.md) — elevator pitch, mental model, status.
- [docs/manifesto.md](./docs/manifesto.md) — why this exists, why now, why this shape.
- [docs/governance.md](./docs/governance.md) — contribution norms, review responsibilities.
- [docs/prd.md](./docs/prd.md) — technical north star: data model, governance machinery, calibration, credit, adversary testbed.
- [docs/seed-topic.md](./docs/seed-topic.md) — first cause for the public instance, starter sub-topics, and rationale.

**Exit criterion:** the docs are stable enough that the testbed builds from them without surfacing further contradictions or underspecified load-bearing points. Adversarial review by capable models acts as a continuous pre-screen; outside-expert and friend review are welcome but not gating.

---

## Phase 1 — Adversary testbed (concurrent with Phase 0)

**Goal:** A simulation harness that runs the full governance regime against a synthetic adversarial population, with published results.

This is the unique technical asset of Anchorage and the cheapest credible artifact we can produce. **The exit from Phase 1 is the user-exposure gate** — no real users meet Anchorage until this phase's exit criterion is met. Phase 1 work runs concurrently with Phase 0 doc work.

**Scope:**

- Real graph schema (claim-graph substrate, multi-scale topic/sub-topic/claim, edges, anchors).
- Real write-path tools (the same tools the eventual public instance will expose).
- Real governance machinery: redundant peer review, calibration batches drawn from validated history, reputation scoring, staking.
- Synthetic contributor population spanning the taxonomy: honest-weak, honest-strong, lazy, hallucinator, strategic adversary, patient adversary, sybil farms, coalitions.
- Parameter sweeps and attack-success-rate measurements.

**Artifact:** the testbed code (open) and a public results post / paper documenting attack success rates against tunable defenses.

**Exit criterion:** governance changes are CI-checked against the adversary suite; the published results survive third-party replication.

---

## Phase 2 — Single-cause public instance

**Goal:** One umbrella cause running on Anchorage with real human contributors, two or three hand-seeded starter sub-topics, and a manuscript projection emerging from the first sub-topic to mature.

**Scope:**

- Auth, identity, per-(cause, sub-topic) reputation.
- The verifiable-anchor write path (PMID/DOI fetch, span verification, refusal of ungrounded citations).
- Frontier surfaces: cause-level (where sub-topics could productively open) and sub-topic-level (specific synthesis gaps).
- Review queue with calibration batches.
- Minimal manuscript projection: outline view tying sections to sub-topic subgraphs.
- Operational tooling: moderation, abuse-flagging, reviewer-fraud detection.

**Exit criterion:** the first sub-topic ships a manuscript projection with named contributors, traceable back to graph nodes, and the testbed catches at least one governance proposal that would have been an attack vector.

---

## Phase 3 — Second cause + protocol hardening

**Goal:** A second umbrella cause running on the same instance, validating that the protocol is cause-agnostic.

**Scope:**

- Sub-topic auto-discovery as a graph-derived feature (proposing tractable scope envelopes from graph state).
- Cross-cause reputation: how (or whether) reviewer credibility transfers between causes.
- Federated read; optional federated write.
- Manuscript-projection improvements: section-level claim provenance, citation export, reviewer comments tied to graph nodes.

**Exit criterion:** the second cause produces an independent manuscript projection without governance regressions on the first.

---

## Phase 4 — Independent fork

**Goal:** At least one institution we don't control runs an independent Anchorage instance with a different cause focus.

**Scope:** documentation, deployment story, governance handoff, federation contract.

**Exit criterion:** the independent instance is producing manuscript projections on its own cause, and protocol changes can be coordinated across instances without breaking either.

---

## What's deliberately *not* on this roadmap

- A token, marketplace, or paid tier.
- Generic chat or freeform-wiki features.
- Auto-merge of contested syntheses (the system surfaces them as `open_question` instead).
- Replacement for journal review or empirical research.
- Promises about specific timelines. Phases happen when their exit criteria are met.

## Status

Phase 0 + Phase 1, concurrent. Design docs are settled and the v0 MCP tool surface is implemented end-to-end as a TypeScript `Server` class — all 17 tools from the PRD, the curator-mediated acceptance path for every proposal kind, the assignment loop (capacity → request → accept → submit), the review path (`cast_review_vote`), and the read-path projections (`query_frontier`, `query_proposals`, `fetch_calibration_batch`, `query_reputation`). The contributor lifecycle composes correctly under test. MCP transport runs the same code paths a real client sees, and the testbed harness drives synthetic archetypes (honest contributors, honest-weak contributors, honest and lazy reviewers, hallucinator, strategic-coalition reviewers) end-to-end through the wired surface — with reputation deltas, verifier rejections, and bias-driven convergence outcomes observable over the wire. The strategic-coalition scenario surfaces an open attack: pure-vote convergence + reputation-against-converged-outcome lets a 2-of-3 biased coalition flip both ledgers against an honest reviewer of well-grounded work. Calibration is wired as the rep-ledger defense (assignment-injected items drawn from validated history, scored on `cast_review_vote` against ground truth) and the testbed shows it costs the coalition rep on bias-misaligned items while crediting the honest reviewer. Calibration-aware convergence weights votes by per-(cause, sub-topic) calibration record and requires both a minimum distinct-reviewer count and a minimum weighted sum; with that flag on, the naive coalition can no longer drive a bias-aligned suppression of a well-grounded excerpt past the threshold — the convergence half of the attack closes against the naive archetype while the rep-ledger half stays inverted, the two layers compounding. Against the calibration-aware variant of the archetype (PRD §Calibration batches — bias narrow enough that the calibration corpus falls entirely outside the bias zone), both defenses go quiet by design — the coalition passes every calibration item, keeps full convergence weight, and reclaims rep dominance as well as suppression. Pool-size scaling closes that vector against any fixed-size coalition where the eligible honest reviewer pool is large enough: at threshold N a coalition of size < N cannot solo-drive convergence on either side, so K+1 honest reviewers outvote a K-member coalition. The minimum K+1-honest-reviewer regime is testbed-handled. The small-sub-topic case where the eligible pool can't furnish K+1 honest reviewers is now closed by stratified-by-history assignment: PRD §Reviewer assignment commits the v0 stratum primitive (vote-pattern co-occurrence cluster), the prefer-not-yet-represented-strata draw rule, and the stratification-degraded flag that tightens convergence thresholds when fewer than N strata are reachable; `request_assignment` and `resolveByConvergence` are wired to that partition. The testbed scenario shows the calibration-aware coalition that bypassed both calibration defenses failing here — once Carol and Dave have built shared vote-history, the cross-stratum draw routes Dave away from the contested proposal, and the bias-suppression vector closes against a 3-reviewer pool too small for K+1 honest. The vote-decorrelating coalition variant — Carol and Dave deliberately disagreeing on every non-bias-zone proposal so their pairwise agreement falls below the cluster threshold — is now also wired as a testbed scenario, and confirms what the v0 cluster primitive cannot catch: the coalition holds singleton strata, the cross-stratum gate has nothing to enforce, both members vote on the contested item, and the bias-suppression vector reopens. The seam is the cluster signal itself: vote co-occurrence is lossy in two opposite directions. Contention-weighted edges are now wired as the first refinement and close the *over-aggregation* failure mode — where unanimous priming history pulls honest reviewers into one cluster and the cross-stratum gate locks them out, while coalition members who skipped priming walk in through the gap. The over-aggregation testbed scenario shows the raw primitive driving the contested item to rejected (honest review strangled), and the contention-weighted companion shows it converging accepted (unanimous-easy priming carries 0 weight, no edges form, all five reviewers vote). Anti-correlation detection is now wired as the second refinement and closes the decorrelating-coalition bypass. The cluster computation treats systematic disagreement as a co-stratum edge alongside systematic agreement: a pair that votes opposite on every shared priming proposal lights up the disagreement metric the agreement metric can't see, gets pulled into one cluster, and the cross-stratum gate trips against the second coalition member when the first has already been routed. The testbed scenario for the decorrelating coalition closes at threshold 1.0 (perfect decorrelation). The mixed-strategy coalition that targets the *gap* between the two thresholds — a 50/50 vote pattern across shared priming, sitting below both — defeats either refinement individually but the *composition* (contention-weighted + anti-correlation, both on at threshold 1.0) closes it: contention weighting reweights the pair-stat so agreement on unanimous-easy items contributes 0 weight and disagreement on contentious items contributes the full weight, collapsing the weighted disagreement ratio back to 1.0. The testbed wires both halves: the mixed-strategy bypass open with each refinement alone, closed under the composition. The remaining seam — a coalition that avoids co-voting on any contentious item, which constrains them sharply (suppressing the contested target inherently creates a contentious item) — has its concrete instance in the multi-proposal paired-decline coalition wired in the paragraphs below: the structural closure (encounter-domain extension via `stratum_include_declines`, paired-decline floor) and the operational closure (divergence-closure archival) both land on it, and cube #3 measures the ASR lift from declines on top of votes.

The decline-pattern projection (PRD §Adversary testbed: "decline-pattern abuse — declining everything outside the adversary's preferred sub-topic") is now wired as a curator-side surface (`server.curator.declinePatterns(cause_id)`), with `min_offers` and `min_rate` filters that the curator picks per their threshold (operationally private). The testbed scenario shows a reviewer who declines everything outside their preferred shape surfacing at the top of the projection; an honest accept-all reviewer ranks at zero and is filtered out at any non-trivial `min_rate`. The divergence-closure mechanism PRD §Reviewer assignment commits is also now wired (`server.curator.archiveStaleProposals({ window_seconds, cause_id? })`): a staged proposal whose most recent vote is older than the window flips to `unresolved-archived`, the terminal status the contracts already declared but no path produced. Never-voted proposals are explicitly excluded — they're unstarted, not divergent. The sweep is idempotent (already-archived proposals are skipped on re-run) and curator-triggered, with the production scheduler kept operationally private.

A first parameter sweep is wired as the Phase-1-exit-criterion infrastructure pattern: a 3D cube over (coalition pattern ∈ {mixed, decorrelated}, anti-correlation threshold ∈ {0, 0.5, 1.0}, contention-weighted ∈ {off, on}) drives twelve it.each cells, each running the same end-to-end scenario through a shared `runDecorrelationScenario` helper and asserting the contested-status outcome. The matrix exercises all four corners of the closure: threshold 0 leaves every cell open; threshold 0.5 catches both patterns; threshold 1.0 closes the decorrelated pattern outright but only catches the mixed pattern under the contention-weighted composition (the matrix cell that was the standalone-test headline). The cube is now layered with an aggregate-attack-success-rate assertion that groups cells by defense config (anti-correlation threshold × contention-weighted) and asserts the per-config ASR — 100% with anti-correlation off, 0% at threshold 0.5, 50% at threshold 1.0 alone (the named gap), 0% under the composition. The aggregate is derived from the static expected-status fields the per-cell tests already validate, so it stays a fast read over locked observations rather than a re-run of the cube. The shape — named defense config to expected ASR, with a coverage invariant that catches uneven attack-pattern counts per group — is the template the four cubes that followed all layer on (cube #2 across the assignment-gate thresholds, cube #3 across the encounter-domain widening, cube #4 across contention-weighting × the paired-decline floor, cube #5 across the difficulty-aware re-baseline, each described below), and cube #5 generalized the single-ASR aggregate to a two-metric (ASR, lockout-rate) split that retrofitted into cube #4.

Honest-weak is wired as the friction-rate archetype — structurally honest-strong with verifier-rejection tolerance, with competence modeled in the content provider rather than the loop. The friction-rate test pins the measurement: the fraction of attempts the verifier refuses is observable from the action log and matches the configured weakness exactly, sitting between honest-strong's 100% pass-through and the hallucinator's 0%. The same cost (a `submit_rejected` action, a burned rate slot, no proposal record) applies — the archetypes differ in intent and fraction, not mechanism.

The sybil-amplified coalition is now wired as the freshness-bypass scenario for stratification: an established Carol+Dave coalition (priming history detected by the cluster signal) recruits Eve, a fresh identity with no shared history. The cross-stratum gate fires correctly against Dave (Carol-already-routed + Carol-Dave co-stratum) but Eve sits in a singleton stratum and the gate has nothing to enforce against her; she walks in and casts the second reject vote. The contested target flips to rejected with three votes — Erin's accept, Carol's reject, Eve's reject — and Dave never votes. The pin is architectural: behavior-dependent defenses (cluster signal, calibration record, reputation) all require accumulated history per identity, so an operator that mints fresh identities and uses each exactly once routes around them by construction. PRD §Identity specs four sybil-resistance layers — binding cost (`attestation_level` on the identity record, `min_attestation_level` gate at every write tool), issuance-frequency cap (below MCP, modeled in the testbed's adversary-budget model as the time primitive on the cost-multiplier), per-identity rate-limit accounting (new `rate_limited` refusal mode at the write surface), cross-cause identity-clustering (curator-side projection parallel to `declinePatterns`) — composing multiplicatively against an adversary budget per the §Adversary testbed adversary-budget model. The binding-cost gate is wired (slice 2, paragraph below); the remaining three layers land in subsequent slices alongside the testbed harness's adversary-budget model that joins the existing parameter-sweep cubes. This scenario is the regression handle on the seam those layers close — the runner mints synthetic identities at attestation 0 and keeps `min_attestation_level=0` (default) so the binding-cost gate stays inert, preserving the scenario's role as the open-seam handle for the layers that haven't landed yet.

Cross-cause identity-clustering is now wired as the fourth of the four sybil-resistance layers (`server.curator.identityClusters({ window_seconds, min_signal })`, parallel to `declinePatterns`). The v0 signal is per-(reviewer pair) count of distinct causes where both reviewers cast votes on the same proposal — honest reviewers typically work in one cause (per-cause reputation), so a pair appearing on shared proposals across multiple causes is the cross-cause behavioral fingerprint a sybil farm working multiple causes lights up and a single-cause coalition does not. Two metrics surface per pair: `cross_cause_count` (the headline signal) and `shared_proposal_count` (visibility tiebreaker — high cross_cause_count with low shared_proposal_count reads differently from concentrated co-occurrence in fewer causes; the curator weighs both). Default `min_signal=2` (cross-cause coordination requires at least 2 causes to be cross-cause); `window_seconds` is the rolling window over vote `created_at`, undefined for all-time. Sort order matches `declinePatterns`'s pattern: signal desc, then secondary metric desc, then ids asc as tiebreaker. Specific signals and thresholds remain operationally private at the production instance per PRD §Identity bullet 4 — methodology is public, tuning is not. The new regression scenario pins both halves of the contract: a sybil pair voting across CRC and AMR surfaces with `cross_cause_count=2` at `min_signal=2` (default), an honest pair voting in one cause only is filtered out by construction; lowering `min_signal=1` brings the honest pair in and confirms the sort order (sybil pair first by `cross_cause_count desc`). Computation walks the global `reviewVotes` store once, builds per-proposal reviewer sets, generates pairwise tuples for proposals with ≥2 reviewers, and accumulates causes/proposals per pair; pair canonicalization (sorted-id tuple) collapses (a,b)/(b,a) duplicates. The projection currently surveys vote co-occurrence; declines are first-class encounters in the cluster-stratification primitive (`stratum_include_declines`) and adding them here is a follow-up refinement that broadens the co-engagement notion without changing the surface shape — the curator's two-metric view scales to per-(cause, identity) decline counts as a third metric whenever the seam needs it.

Per-identity rate-limit accounting is now wired as the third of the four sybil-resistance layers (`rate_limit_actions_per_epoch` and `rate_limit_epoch_seconds`, both default Infinity → gate inert). The cap is single-bucket across all write tools — total throughput per identity per epoch — so the cost-multiplier reads as the per-sybil-throughput axis directly: at K sybils with cap T, the coalition's per-epoch budget is K × T. The shared `accountWriteAction` helper fires at the same 13 write-tool seams the binding-cost gate covers (atomically check-and-increment the per-(identity, epoch) counter, throw `ServerError('rate_limited', ...)` when the cap is reached). Refusal mode is the new `rate_limited` ServerErrorCode parallel to `unauthorized` and `not_found` — the recovery path is "wait for the next epoch," not "obtain a higher attestation" or "no work available," and conflating these would mislead a contributor whose throttling is transient. The new code is justified by the distinct recovery path and PRD §Identity's architectural commitment; `errors.ts` notes that adding a code is breaking-change territory and the comment block on the new entry names the rationale (the rate-limit refusal's recovery path is genuinely distinct, so reusing one of the existing three would have surfaced the same drift on the contributor side as the testbed). Counter consumption is positioned in the call order between the binding-cost gate and the tool-specific business logic — unauthorized callers don't burn budget; valid-input-but-business-logic-failed calls do (an adversary can't probe the system with bogus business inputs without paying for each). The store keeps a single record per identity (`{ epoch, count }`) that the helper resets lazily on epoch boundary rather than maintaining one entry per (identity, epoch) tuple — historical per-epoch counts aren't needed for enforcement; slice 4's curator-side identity-clustering projection is where historical surveillance signals surface, not enforcement. Epochs are wall-clock windows (`floor(now_ms / (epoch_seconds * 1000))`), not sliding windows; `FakeClock.advance(ms)` is what testbed scenarios use to cross epoch boundaries deliberately. The new regression scenario pins both halves of the contract: 3 actions land at cap=3, the 4th refuses with `rate_limited`, a *different* write tool also refuses with `rate_limited` (proving the cap is global rather than per-tool), the clock advances past the epoch boundary, and the same identity acts again successfully — counter reset is observable end-to-end.

The binding-cost gate is now wired as the first of the four sybil-resistance layers (`min_attestation_level`, default 0). The IdP records `attestation_level` on the identity at mint, opaque to the server (the server gates on threshold; it does not interpret what the level *means* — that's the IdP's job); the gate fires at every write tool that resolves a caller — `set_capacity`, `request_assignment`, `accept_assignment`, `decline_assignment`, `submit_assigned_proposal`, all `propose_*` tools, and `cast_review_vote` — refusing with `unauthorized` mode rather than the rep gates' `not_found` opacity. The mode choice is load-bearing: the binding-cost mismatch is identity-level (a stable property of the IdP-issued credential) and there's no opaque-refusal value to protect (the adversary already knows their own attestation level), so the refusal accurately names the mismatch instead of masking it as the rep gates do for their thresholds. Read-path tools (`query_frontier`, `query_proposals`, `query_reputation`, `fetch_calibration_batch`) are not gated — read access doesn't multiply the per-identity action budget the gate is defending against, and PRD §Identity scopes the gate to the write surface explicitly. A single shared `requireMinAttestation` helper is what fires at every write seam, so coverage extends uniformly without per-tool branching; the new regression scenario (in `testbed.test.ts`) walks a representative cross-section of seams — capacity, assignment pull, contributor-initiated propose, contributor-initiated review — and confirms `unauthorized` at each, then mints an above-threshold identity and confirms the same seams pass with the gate inert. Default 0 leaves the gate inert and preserves all existing scenarios' behavior. The cost-multiplier the gate consumes is measured in slice 5 by the testbed harness's adversary-budget model (PRD §Adversary testbed: adversary budget model) — at K sybils the coalition pays K × per-sybil binding cost upfront, and identities below threshold cannot accrue any per-identity activity that behavior-based defenses (cluster signal, calibration record, reputation gates) would later engage on.

Patient adversary is wired as a drift-bandwidth archetype — a calibration-aware biased reviewer running solo, priming through honest accepts on well-grounded excerpts and passes on the anchor calibration corpus, then drifting on a single contested target. The scenario stacks the strongest current defenses (calibration injection + calibration-aware convergence) and confirms the convergence half holds: the contested target converges to accepted because two honest reviewers reach the distinct-count + weighted-sum gate before the lone biased vote can find a partner.

Two-component reputation bookkeeping is wired as the first half of the patient-adversary defense: every reputation event moves a slow-decay `demonstrated`-competence component and a fast-decay `recent`-activity component together, and `query_reputation` decays each component on read per its own half-life (PRD §Reputation). Half-lives are testbed-tuned config — `demonstrated_half_life_seconds` and `recent_half_life_seconds`, both defaulting to `Infinity` so existing scenarios (no time advanced) see identical pre-decay numbers on both fields and stay unaffected. The patient-adversary bookkeeping scenario is the regression handle: immediately post-drift `demonstrated == recent`, but advancing the clock past several recent half-lives drains `recent` toward zero while `demonstrated` is preserved.

The recent-activity *assignment gate* is now wired as the second half (`assignment_min_recent`, default 0). At `request_assignment`, callers with rep entries in the requested cause whose max decayed `recent` falls below the threshold are refused with `not_found`. Fresh reviewers (no rep entries) bypass the gate by construction so first-assignment bootstrap stays open; contributor-initiated voting (`cast_review_vote` with no `assignment_id`) bypasses too, so a drained-but-honest contributor has the named recovery path. The patient-adversary gate scenario closes the loop the bookkeeping scenario named: Carol primes + drifts on a first contested target, the clock advances past six half-lives, Carol's recent falls below threshold, a fresh contested target is staged — and her second drift attempt fails at `request_assignment` time. Drift bandwidth tightens from "many" (cumulative buffer) to 1 (the drift she landed before the quiet window). `FakeClock.advance(ms)` was added so testbed scenarios travel time deliberately on top of the per-call auto-tick.

The demonstrated-component *eligibility-tier gate* is now wired as the companion gate (`assignment_min_demonstrated`, default 0). Same seam as the recent gate (`request_assignment`), opposite null-policy: callers with no rep entries in the requested cause **fail** the gate when threshold > 0 — the demonstrated tier is "have you proven yourself yet?" and an unproven identity is by construction not in the pool. Bootstrap is the contributor-initiated path (direct `propose_*` and `cast_review_vote` without `assignment_id`), which earns rep without traversing the gate; once the caller's max decayed `demonstrated` clears the threshold, `request_assignment` opens. The eligibility-tier scenario pins the shape: Eve mints fresh, declares cause-level capacity, is refused at `request_assignment` (`not_found`), bootstraps via two accurate contributor-initiated accept votes alongside Erin (also fresh, also bootstrapping symmetrically), and graduates to her first assignment. The two gates compose at the same seam — recent fires against once-active-now-quiet (the patient-adversary signature), demonstrated fires against never-proven (the sybil-amplified-coalition signature) — and together they constitute the visible-history cost a fresh-identity coalition pays before being drawn for review. PRD §Identity's three remaining sybil-resistance layers (issuance-frequency cap, per-identity rate-limit accounting, cross-cause identity-clustering) land in subsequent slices alongside the testbed harness's adversary-budget model; the binding-cost gate that's already wired (paragraph above) composes with these two rep gates as a separate identity-level layer that fires before any per-identity activity accrues, while the rep gates fire on accumulated activity once it does.

The second parameter sweep cube is now wired across these two gates. Eight cells over (attack pattern ∈ {patient, sybil}, `assignment_min_recent` ∈ {0, 0.5}, `assignment_min_demonstrated` ∈ {0, 1.5}) drive `it.each` through two shared scenario runners — `runPatientAdversaryGateScenario` and `runSybilAmplifiedGateScenario` — and assert per-cell `attack_succeeded` and `false_positive_lockout` under cube #5's two-metric template. Both runners respect what makes their archetypes legible: each runs a contributor-initiated bootstrap rotation that pairs honest reviewers across three convergences on disjoint excerpts so each honest reviewer ends with `demonstrated=2.0` (clears any threshold ≤ 1.5) before the assignment-driven phase begins, with each pair sharing exactly one bootstrap proposal so the honest pool stays singleton in the cluster signal (`stratum_min_shared_proposals: 2` is the floor the rotation sits below). The patient runner uses Carol+Erin+Frank in the rotation (Carol later drifts as the adversary); the sybil runner uses Erin+Frank+George as the honest pool while Carol+Dave coalition-prime separately on three other excerpts and Eve recruits fresh. Without this bootstrap the demo gate would conflate fresh-coalition with the legitimate honest pool by closing both at the cold-start, and the cube would mis-read lockout-driven closures as honest defense. The aggregate groups by defense config and reads both metrics: (recent=0, demo=0) at (ASR=100%, lockout=0%); (recent=0.5, demo=0) at (ASR=50%, lockout=0%) — recent gate closes patient by honest defense, sybil bypasses through the fresh-reviewer recent-gate null-policy; (recent=0, demo=1.5) at (ASR=50%, lockout=0%) — demo gate is inert against patient (Carol's demonstrated buffer holds), closes sybil by Eve's null-policy while the honest pool's bootstrap demonstrated clears the gate, Carol gets routed and rejects, Dave is cross-stratum-gated, and Erin+Frank's accepts converge contested across singleton strata; (recent=0.5, demo=1.5) at (ASR=0%, lockout=0%) — composition closes both attacks by honest defense at both metrics. The orthogonal-defenses headline reads cleanly under the two-metric template — every closure cell is honest defense, no lockout-driven cell pretends to defense success.

The multi-proposal coalition seam — the "avoid co-voting on any contentious item" workaround the mixed-strategy headline pinned by name — is now wired as a standalone scenario. Two coalition members each solo-reject a *different* contested target and decline the partner's, so the cluster signal stays silent by construction (zero shared voting history → no edge metric has data to fire) and convergence stalls at 1 accept + 1 reject on each target. The wired closure stack runs at full strength against this — strongest cluster signals on (contention-weighted, anti-correlation 1.0, agreement 1.0), default assignment gates open — and fires nothing. The terminal outcome is the divergence-closure sweep: both targets archive as `unresolved-archived`, which from the proposer's perspective is the suppression-by-archival result the coalition was driving toward. The decline-pattern projection's *default* small-sample floor (`min_offers=3`) hides the coalition entirely at the 2-target scale; surfacing them requires the curator to drop the threshold to the small-N regime, which in the same act surfaces every casual decliner alongside them — the visibility surface exists but pays its own false-positive cost on the small-pool case.

The decline-pattern → assignment-gate is now wired as the operational form of the curator-side `declinePatterns` projection. PRD §Capacity and assignment commits the gate as the assignment-time enforcement primitive on the same signal: `assignment_max_decline_rate` (default 1.0, gate inert) and `assignment_decline_min_offers` (small-sample floor) consumed at `request_assignment`, refusal as `not_found` so the contributor-facing surface stays structurally indistinguishable from "no work available," and the contributor-initiated path (`cast_review_vote` without `assignment_id`) bypasses by construction so a contributor whose decline rate has spiked retains the recovery path declines are non-punitive precisely *to* preserve. The gate's mechanics are pinned standalone: above-threshold + above min_offers refuses; above-threshold but below min_offers bypasses; fresh reviewer (zero offers) bypasses; the contributor-initiated path bypasses. The numerator and denominator the gate consumes match the curator-side `declinePatterns` projection byte-for-byte — same per-cause cumulative rate — so the assignment-time and curator-side surfaces operate on a single signal.

The gate is *operational* — a curator-tunable enforcement primitive on a known abuse signal — not a *structural* closure of the multi-proposal seam. The gate fires at `request_assignment` time and uses cumulative state, so it can refuse a caller's *next* request after their decline rate exceeds threshold but does not undo declines that already landed. At the minimal seam (2 contested targets, 1 honest reviewer), the coalition's lone-rejects on each target land before the gate could fire, both targets stall at 1 accept + 1 reject, and the divergence-closure sweep archives them regardless of the gate. The gate stays useful as the throttle a curator uses against decline-mediated abuse over a longer time-horizon than a single batch (where the gate's throughput cap binds — coalition size × declines-per-identity is the budget a real-world adversary works under).

The structural defense for paired-decline is the cluster signal extended to include declines as encounters (`stratum_include_declines`, default off). PRD §Reviewer assignment commits the widening: the encounter domain becomes `{accept, reject, decline}`, so "Carol votes A and declines B; Dave declines A and votes B" reads as 100% pair-disagreement on two shared encounters under the existing anti-correlation primitive — the seam evades the vote-only cluster by construction, and the wider domain closes it at the same primitive that already catches vote-only decorrelating coalitions. A real vote takes priority over a decline on the same `(reviewer, proposal)` pair, and decline-involved encounters under contention-weighting count at full weight (declines are inherently informative; the per-proposal accept/reject tally would underweight them by construction). The structural-effect scenario pins both halves: with the knob off, paired-decline encounters yield zero shared vote-history, no edge forms, the cross-stratum gate has nothing to enforce against, and the partner walks into the contested target normally; with the knob on, the cluster forms after the priming, the cross-stratum gate trips, and the partner's request_assignment fails with `not_found`. The whole-scenario closure landed alongside it as a multi-round companion: paired-decline priming on two excerpts builds the cluster *before* the contested target is staged, so when the suppression round begins the gate is already armed. The adversary-budget model is now wired as the testbed-side fiction PRD §Adversary testbed §Architecture commits — slice 5 of the identity-cost design pass and the closer on the four-layer sybil-resistance architecture. The `AdversaryBudget` primitive holds a numeric budget B and operationalizes the three deductions the four layers compose multiplicatively against: `tryMint(epoch)` deducts α from B (binding-cost layer; mirrors the server-side `min_attestation_level` gate which fires on the same α) and refuses with the `issuance_cap` failure mode when the per-epoch cap is reached (issuance-frequency-cap layer; testbed-only, no server surface — the IdP refuses upstream of any cost being charged, so budget is unchanged on `issuance_cap` refusals); `tryAct(identity, epoch)` accounts the per-(identity, epoch) throughput cap T (mirrors `accountWriteAction` on the testbed side so the harness reads coalition throughput as K × T directly). Failure modes on `tryMint` are distinct: `budget` is "the operator cannot afford another sybil at any rate" (α > remaining B), `issuance_cap` is "the operator could afford it but the IdP refuses to mint another in this epoch" — the distinction matters when sweeps measure ASR as a function of coalition-affordable-identities-per-epoch, since the budget axis caps the total K and the issuance cap shapes its distribution across epochs. Epochs are passed in by the caller rather than derived from a clock — the testbed knows what epoch each action sits in by construction, and keeping the primitive arithmetic-only avoids coupling it to the server's `clock.now()` tick semantics. The new regression scenario pins each deduction directly: budget drains α-by-α across mints, the same-epoch second mint refuses with `issuance_cap` (budget unchanged), the next-epoch mint succeeds, the budget-exhausted mint refuses with `budget` (distinct from `issuance_cap`), per-(identity, epoch) action cap fires after T actions and resets on epoch boundary, identities have independent per-epoch counters so the K × T composition holds. Default action cap is Infinity (gate inert) for sweeps that hold throughput unbounded and isolate the identity-creation axis. The clustering projection (slice 4) costs the adversary nothing in the primitive — that layer surfaces at the curator surface as visibility, not budget consumption.
