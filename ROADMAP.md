# Roadmap

Anchorage is built phase by phase, with each phase producing a discrete artifact that's useful on its own and informs the next. This file is honest about what's planned, what's hand-waved, and what's load-bearing for everything else.

The roadmap is a living document. Phases will move; some will split or merge. What stays stable is the *order of dependencies*: each phase produces something the next phase needs, and we don't skip ahead.

## Phase 0 â€” Design (closed 2026-05-14)

**Goal:** A coherent design document set that the next person to read can decide *in ten minutes* whether they want to be involved.

**Concurrency.** Phase 0 and Phase 1 ran concurrently. Doc work and testbed work co-evolved, with the docs-never-drift discipline (see [CLAUDE.md](./CLAUDE.md)) keeping them in lockstep â€” every commit that touched a contract updated both. Phase 0 finished when the docs stopped moving under their own pressure; the testbed has been building from them through several deep-loop cube cycles without surfacing further design contradictions or underspecified load-bearing points.

**Artifacts:**

- [docs/governance.md](./docs/governance.md) â€” contribution norms, review responsibilities.

- [docs/seed-topic.md](./docs/seed-topic.md) â€” first cause for the public instance, starter sub-topics, and rationale.

**Exit criterion (met):** the docs are stable enough that the testbed builds from them without surfacing further contradictions or underspecified load-bearing points. Adversarial review by capable models acted as a continuous pre-screen; outside-expert and friend review were welcome but not gating.

## Phase 1 â€” Adversary testbed (closed 2026-05-14)

**Goal:** A simulation harness that runs the full governance regime against a synthetic adversarial population, with a published results snapshot.

This is the unique technical asset of Anchorage and the cheapest credible artifact we can produce. **The exit from Phase 1 is the user-exposure gate** â€” no real users meet Anchorage until this phase's exit criterion is met. Phase 1 work ran concurrently with Phase 0 doc work.

**Scope:**

- Real graph schema (claim-graph substrate, multi-scale topic/sub-topic/claim, edges, anchors).
- Real write-path tools (the same tools the eventual public instance will expose).
- Real governance machinery: redundant peer review, calibration batches drawn from validated history, reputation scoring, staking.
- Synthetic contributor population spanning the taxonomy: honest-weak, honest-strong, lazy, hallucinator, strategic adversary, patient adversary, sybil farms, coalitions â€” both scripted (the fast loop) and model-backed (the deep loop, via `runLlmAgent`).
- Parameter sweeps and attack-success-rate measurements (the cube layer in `testbed.test.ts` for scripted populations and `run-deep-loop-cube.ts` for model-backed populations).

**Artifact:** the testbed code (open, AGPL-3.0) and a published results snapshot documenting attack success rates against tunable defenses ([docs/phase1-results.md](./docs/phase1-results.md), CC BY-SA 4.0).

**Exit criterion (met):** governance changes are CI-checked against the adversary suite (the cube layer is the regression handle for every named axis in PRD Â§What's deliberately not specified, with the harness pairs in `population-loop.test.ts` pinning byte-for-byte deltas of every closure-stack knob), and a published results snapshot exists at the cited artifact. The original ROADMAP language called for "third-party replication" of the published results; this was a category error borrowed from conventional scientific publishing. Anchorage's testbed is by-construction simulatable: cassettes pin every byte across the wire, the model is a published API, the code is open. A third-party rerun is *literally the same operation* as an in-house rerun â€” the third party has no epistemic advantage except "they ran it instead of us". The load-bearing question is sweep coverage, not who runs the sweep; further coverage is incremental work in the same testbed and does not gate Phase 2 ([docs/phase1-results.md Â§Methodology](./docs/phase1-results.md) carries the reproduce-it instructions for anyone â€” in-house or third-party â€” who wants to extend the sweep).

## Phase 2 â€” Single-cause public instance (opened 2026-05-14)

**Goal:** One umbrella cause running on Anchorage with real human contributors and a manuscript projection emerging from the first sub-topic to mature. The instance opens with a single hand-seeded sub-topic and seats the second/third from the [`docs/seed-topic.md`](./docs/seed-topic.md) shortlist on a maturity-or-contributor signal â€” the first sub-topic reaching an active node, or enough contributors that a new envelope won't strand reviewers â€” because concentration, not breadth, is what lets the redundant-review mechanic work at low contributor count: two contributors split across three sub-topics may never overlap, and nothing leaves `staged` without an independent reviewer. The two-or-three starter set remains the Phase 2 end-state, not the launch state; the shortlist stays pre-vetted and on-deck.

**Scope:**

- Auth, identity, per-(cause, sub-topic) reputation.
- The verifiable-anchor write path (PMID/DOI fetch, span verification, refusal of ungrounded citations).
- Frontier surfaces: cause-level (where sub-topics could productively open) and sub-topic-level (specific synthesis gaps).
- Review queue with calibration batches.
- Minimal manuscript projection: outline view tying sections to sub-topic subgraphs.
- Operational tooling: moderation, abuse-flagging, reviewer-fraud detection.

**Slice plan:** The phase decomposes into a sequence of shippable slices. Each lands doc-then-code per [CLAUDE.md Â§Conventions for working in this repo](./CLAUDE.md#conventions-for-working-in-this-repo); each preserves the simâ‰¡prod invariant â€” the testbed continues to use the deterministic in-memory primitives (`FakeVerifier`, in-memory `Store`, harness-minted identities), and the production code path acquires its real counterpart at the same surface, with no `if (sim) ...` branching. Slices stack in dependency order; later slices may be reordered if mid-phase observations argue for a different sequence, but any contract change still lands doc-and-code in the same commit.

1. **Live-fetch verification (closed 2026-05-14).** Replace the `StructuralVerifier` placeholder with a real verifier hitting NCBI E-utilities (PMID) and Crossref (DOI); URL anchors refused by default per PRD Â§Verification engine (URL-anchors are second-class in v0, opt-in via `reject_url: false`). Span verification against the fetched source; content-addressed anchor hashing (sha256 over normalized content); shared `normalizeForSpanMatch` covering whitespace, smart-quote fold, en-/em-dash/minus and ellipsis folds, used by both the live verifier and the testbed `FakeVerifier`. The testbed continues to use `FakeVerifier` with seeded source maps â€” cassette-deterministic CI requires no production verifier in the loop. Re-verification (the path that transitions anchors to `unresolvable` on content drift) is a follow-up alongside slice 7's operational tooling; the verifier-side primitive is in place. Wiring `LiveFetchVerifier` as the default `Server` verifier is the production runtime in slice 4c (`run-prod.ts` constructs `Server` with `verifier: new LiveFetchVerifier()`).

2. **Persistence (closed 2026-05-14).** Replace the in-memory `Store` with a SQLite-backed implementation. The `Store` surface â€” `MapLike<K, V>` (`get` / `set` / `values` / `entries` / `size`) over the primary-record collections (identities, agent credentials, agent-credential secrets, causes, sub-topics, proposals, nodes, edges, review votes, assignments, capacities, reputations, calibration records, verified refs, rate limits â€” the `agent-credential secrets` index lands in slice 3b's bearer-token model) â€” is now an explicit interface in `packages/server/src/store.ts`; both `MemoryStore` (testbed default) and `SqliteStore` (production-runtime default; wired by slice 4c) satisfy it. `SqliteStore` is built on Node's built-in `node:sqlite` (Node â‰¥24 is already the engine floor, so no new native dependency); each collection is a single `(key TEXT PRIMARY KEY, value TEXT)` table holding JSON-encoded records, with iteration ordered by `rowid` to match JavaScript `Map`'s insertion-order semantics. Single-instance suffices for the single-cause v1; a Postgres path lands later if multi-instance becomes necessary before Phase 4. The testbed continues to use the in-memory store (cassettes need deterministic id ordering and zero-disk I/O), and a parity test (`packages/server/src/sqlite-store.test.ts`) drives the same end-to-end Server scenario through both backends and asserts byte-identical graph state plus close/reopen durability against an on-disk file â€” the simâ‰¡prod invariant for the storage backend, in code. The persistent-data layout is JSON-in-text rather than structured columns because the Server's only access pattern against the store is by primary key plus full-scan iteration; richer columnar projections (frontier queries, identity clustering) are server-side derivations from iterated state and would be served by separate materialized projection tables alongside, not by reshaping the primary records. PRD has no commitments about the storage backend; the slice introduces no contract change.
